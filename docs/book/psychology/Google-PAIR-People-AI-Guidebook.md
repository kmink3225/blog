## Chapter 1: User Needs + Defining Success

### Overview

AI 제품 설계의 기초: 어떤 문제가 AI에 적합한지 판별하고, 자동화와 증강 중 선택하며, 장기적 사용자 이익을 최적화하는 보상 함수를 설계한다.

> "Even the best AI will fail if it doesn't provide unique value to users."

### 1. Find the Intersection of User Needs & AI Strengths

**AI가 더 나은 경우:**

- 사용자별 개인화된 콘텐츠 추천
- 미래 이벤트 예측 (가격 추세 등)
- 개인화를 통한 경험 개선
- 다양한 맥락의 자연어 처리
- 대규모 엔터티 인식 (얼굴, 객체)
- 진화하는 저빈도 이벤트 탐지 (사기 탐지)
- 도메인 특화 에이전트 경험 자동화 (호텔 예약)
- 정적 인터페이스보다 효율적인 동적 콘텐츠 표면화

**AI가 적합하지 않은 경우:**

- 핵심 경험의 예측 가능성 유지가 필요할 때
- 정적/표준화된 정보 제공
- 오류 비용이 극히 높은 상황
- 완전한 투명성이 요구되는 경우
- 시장 출시 속도가 최우선인 프로젝트
- 사용자가 수동 제어를 명시적으로 선호하는 작업

**핵심 질문 리프레이밍:** "AI를 쓸 수 있는가?" 대신 "이 문제를 어떻게 풀 수 있고, AI가 고유한 가치를 줄 수 있는가?"

**핵심 실무:** 기존 워크플로를 매핑하여 개선 기회를 식별한다. "Wizard of Oz" 테스트로 AI 동작을 수동 시뮬레이션하여 가정을 검증한다.

### 2. Assess Automation vs. Augmentation

**Automation (자동화):** 작업을 사용자 책임에서 완전히 제거한다.
**Augmentation (증강):** 작업 수행에 대한 인간의 능력을 강화한다.

**자동화 적합 조건:**

- 사용자가 해당 작업에 대한 지식이나 능력이 부족
- 지루하고, 반복적이고, 어색하거나 위험한 작업
- 효율성 향상이 중요
- 개인적 투자가 필요 없는 작업

*항상 인간 감독 옵션을 포함한다:* 미리보기, 편집, 테스트, 실행 취소 기능.

**증강 적합 조건:**

- 사람들이 작업 자체를 즐기는 경우
- 개인적 책임이 사회적/감정적 가치를 지닐 때
- 리스크가 높을 때 (신체적, 감정적, 재정적)
- 전달하기 어려운 구체적 선호가 있을 때
- 창작적 표현이나 숙달이 관련될 때

**리서치 질문:**

- "이 작업에 대해 새 동료에게 가장 먼저 무엇을 가르치겠는가?"
- "그 행동을 얼마나 자주 수행하는가?"
- "인간 보조원에게 어떤 업무를 위임하겠는가?"

### 3. Design & Evaluate the Reward Function

보상 함수는 AI가 "올바른" 것과 "틀린" 것을 결정하는 수학적 시스템이다.

**이진 분류 결과:**

- True Positives: 올바른 양성 예측
- True Negatives: 올바른 음성 예측
- False Positives: 잘못된 양성 예측
- False Negatives: 잘못된 음성 예측

**가중치가 중요하다:** 오류 비용은 맥락에 따라 다르다. 화재 감지의 false negative는 치명적이지만, 노래 추천 실패는 사소하다.

**Precision vs. Recall 트레이드오프:**

- **Precision:** 출력이 올바를 확신. 높은 정밀도 = 적은 false positive, 많은 false negative
- **Recall:** 모든 관련 결과가 포함될 확신. 높은 재현율 = 더 많은 true positive 포착, false positive 포함

**보상 함수 평가:**

- 포괄성 평가: 인구통계/맥락에 걸쳐 공평한 경험 보장
- 시간에 따른 모니터링: 첫날뿐 아니라 100일째, 1000일째 경험 고려
- 함정 상상: "이 보상 함수가 완벽히 최적화되면, 그 결과가 사용자와 사회에 도움이 되는가?"
- 부정적 영향 대비: 모니터링 주기 설정, 임계값 알림, 소셜 채널 모니터링

**팀 정렬 프레임워크:**

> "만약 [특정 성공 지표]가 [기능]에서 [임계값] 이하/이상이 되면, 우리는 [특정 조치]를 취한다."

## Chapter 2: Explainability + Trust

### Overview

AI 예측을 설명하고 적절한 사용자 신뢰를 구축하는 방법. 설명 가능성과 신뢰는 본질적으로 연결되어 있다.

### 1. Help Users Calibrate Their Trust

사용자는 맹목적 의존이나 과도한 회의가 아닌, 적절하고 조건적인 신뢰를 개발해야 한다.

**데이터 소스 명확화:**

- 수집되는 데이터의 범위 설명
- 도달 범위 명확화 (개인화 vs. 집계)
- 사용자가 제거하거나 리셋할 수 있는 데이터 지정
- 시스템에서 중요 정보가 누락될 때 인식하도록 지원

**설명을 사용자 행동에 연결:**

설명은 사용자 상호작용에 대한 응답으로 보여줄 때 가장 효과적이다. 명확한 인과관계를 만든다.

**상황적 리스크 고려:**

높은 리스크 시나리오는 더 상세한 설명을 요구한다.

### 2. Calibrate Trust Throughout the Product Experience

**처음부터 신뢰 구축:**

- 기능과 한계를 명확히 전달
- 친숙한 디자인 요소 강조
- 온보딩 중 신뢰할 수 있는 제3자 소스 참조

**초기 신뢰 성장:**

- 개인정보/보안 설정 명시적 공유
- 저위험 탐색을 위한 "샌드박스" 경험 제공
- 선호도 및 피드백 메커니즘을 통한 사용자 제어 활성화

**신뢰 유지:**

- 신뢰가 성장함에 따라 점진적으로 자동화 증가
- 권한/설정에 대해 지속적으로 소통
- 맥락 변경 시 선호도 상기

**신뢰 회복:**

- 오류에 대한 복구 계획을 사전에 설명
- 문제 발생 시 즉각 대응
- 재발 방지 경로 제공

### 3. Optimize for Understanding

많은 AI 시스템은 완전히 설명하기 어려운 확률적 추론을 포함한다. 해결책은 포괄적 투명성이 아니라 전략적 부분 설명이다.

**설명 유형:**

- *일반 시스템 설명*: 전체 동작, 데이터 유형, 최적화 목표
- *특정 출력 설명*: 특정 인스턴스에서 특정 예측이 발생한 이유

**모델 신뢰도 표시 방법:**

- 범주형 (High/Medium/Low)
- N-best 대안 (상위 3~4개 옵션)
- 수치 백분율 (적절한 맥락과 함께)
- 데이터 시각화 (오차 막대, 음영 범위)

**예시 기반 설명:** 결정과 관련된 학습 세트 예시를 보여주어 직관 개발 지원.

**상호작용 기반 설명:** "what if?" 시나리오를 통한 시스템 경계 및 로직 발견.

### 4. Manage Influence on User Decisions

**신뢰도를 보여줄 조건:**

- 사용자 의사결정에 의미 있는 영향을 줄 때만
- 오해의 소지가 없을 때
- 사용자가 올바르게 해석할 수 있을 때

**신뢰도를 보여주지 않을 조건:**

- 결정에 대한 영향이 최소일 때
- 잘못된 예측에 대한 과잉 신뢰를 유발할 수 있을 때

## Chapter 3: Mental Models

### Overview

사용자가 AI 시스템을 이해하는 방식과 정확한 기대치 설정의 중요성. 멘탈 모델은 사람들이 제품을 이해하고 상호작용 결과를 예측하는 방식을 나타낸다.

### 1. Set Expectations for Adaptation

AI 시스템은 적응하고 개인화되어 정적 제품과 다르다.

**기존 멘탈 모델 식별:**

- 사용자가 현재 문제를 어떻게 해결하는지 고려
- 초보 사용자가 사용하는 단계별 프로세스 검토
- 다른 사용자 간 방법의 일관성 평가
- AI 제품으로 전이될 수 있는 멘탈 모델 평가

### 2. Onboard in Stages

**메시징 가이드라인:**

- 기능과 한계를 사전에 명확히 진술
- 가치를 보여주는 구체적 예시 제공
- 사용자 피드백을 통한 개선을 설명
- 피드백 제공의 이점 명시

**메시징 프레임워크:**

> "이것은 [제품/기능]이며, [핵심 이점]으로 당신을 도울 것이다. 현재 [주요 한계]는 할 수 없다. 시간이 지나면 더 관련성 있게 변할 것이다. [시스템을 가르치는 사용자 행동]으로 더 나아지도록 도울 수 있다."

**실험을 위한 설계:**

- 온보딩을 짧게 유지하여 즉시 사용 가능하게
- 저위험, 가역적인 초기 행동 제안
- 초기 실험이 영구적으로 결과를 결정하지 않을 것임을 안심

### 3. Plan for Co-Learning

**피드백 유형:**

- **암묵적 피드백**: 사용자 행동이 시간에 따라 AI를 개선. 이용약관에서 추적 신호 공개
- **명시적 피드백**: 의도적 사용자 입력. 가능하면 구체적 영향과 구현 일정 설명

**우아하게 실패하기:**

- 첫 실패를 피드백 관계 구축의 기회로 활용
- AI가 요청을 완료할 수 없을 때 비AI 기본 경험 제공

### 4. Account for Human-Like Interaction Expectations

**의인화 도전:**

인간처럼 설계된 제품(음성 비서, 대화형 인터페이스)은 반사적으로 인간 특성 추론을 유발한다.

**핵심 요구사항:** 모든 사용자 인구통계와 능력 수준에 걸쳐 알고리즘적 본질과 한계를 명시적으로 전달한다.

**인간화와 정확성의 균형:**

- 인간화 수준을 실제 기능과 인간 인식의 정렬 정도에 맞춤
- 1인칭 언어("I")는 직관적으로 대화를 안내하지만, 거의 완벽한 NLP 기대를 유발
- 개방형 대화보다 어휘 단순성(stop, go) 강조

## Chapter 4: Feedback + Control

### Overview

사용자 피드백과 제어 메커니즘은 AI 모델 출력과 사용자 경험 개선에 중요하다.

### 1. Align Feedback with Model Improvement

**암묵적 피드백:**

앱 열기 시간, 추천 수락/거부, 정기적 사용 패턴 등 제품 로그의 사용자 행동 데이터.

- 어떤 데이터가 수집되고 어떻게 이익이 되는지 사용자에게 알림
- 행동 로깅 옵트아웃 허용

**명시적 피드백:**

평점, 좋아요/싫어요, 설문 응답, 텍스트 코멘트, 분류 평가.

- 명확하고 이해하기 쉬운 질문 사용
- 상호 배타적이고 전체를 포괄하는 옵션 보장
- 맥락과 톤에 적합한 문구 사용

**이중 피드백 해석:**

- 모든 사용자 행동이 유사 콘텐츠 욕구를 나타내는 것은 아님
- 암묵적 신호가 모델 튜닝에 미치는 영향 고려

### 2. Communicate Value & Time to Impact

**피드백 동기 유형:**

| 유형 | 장점 | 단점 |
|---|---|---|
| 물질적 보상 (현금) | 직접적, 양 증가 | 장기 비용, 내재적 동기 저하 |
| 상징적 보상 (뱃지, 지위) | 저/무비용 | 인식 의존, 권력 불균형 |
| 개인적 유용성 (개인화) | 네트워크 효과 불필요 | 개인정보 우려 |
| 이타주의 (공정성) | 정직한 피드백 장려 | 문화에 따라 다름 |
| 내재적 동기 (자기표현) | 참여 즐거움 | 사회적 바람직성 편향 |

**피드백 영향 설명 (최소 → 최대 구체성):**

1. "피드백 감사합니다" (영향 시점/범위 없음)
2. "감사합니다! 피드백이 미래 추천 개선에 도움이 됩니다" (넓은 범위)
3. "감사합니다! 당신의 미래 추천을 개선하겠습니다" (개인 맞춤)
4. "감사합니다! 다음 추천에 언덕을 포함하지 않겠습니다" (구체적 카테고리)
5. "감사합니다. 추천을 업데이트했습니다. 확인해보세요" (즉시, 변경 시연)

### 3. Balance Control & Automation

핵심 원칙: "사용자가 출력을 필요에 맞게 조정하고, 편집하거나, 끌 수 있도록 허용한다."

**사용자가 제어를 유지하고 싶을 때:**

- 작업을 즐길 때
- 개인적 책임이 중요할 때
- 리스크가 높을 때
- 개인적 선호가 복잡할 때

**사용자가 자동화를 수용할 때:**

- 작업을 완료할 능력이나 시간이 부족할 때
- 불쾌하거나, 안전하지 않거나, 즐거움 없이 큰 노력이 필요할 때

**편집 가능성 제공:**

- 시간에 따라 이전 피드백 조정
- 기본 비개인화 버전으로 리셋
- 선호가 변할 때 선택 업데이트
- 필요 변경 시 이전 피드백 삭제

## Chapter 5: Data Collection + Evaluation

### Overview

AI 시스템의 학습 데이터 소싱과 평가. 데이터 품질은 모델 개발보다 투자가 적은 경우가 많아 "data cascades"(학습 데이터와 실제 데이터 간 불일치의 지연된 결과)를 초래한다.

### 1. Plan to Gather High-Quality Data from the Start

고품질 데이터의 특성:

- 실세계 현상을 정확히 대표
- 책임있는 수집, 저장, 사용
- 재현성과 유지보수 가능성
- 재사용 가능성
- 경험적/설명적 힘

### 2. Translate User Needs into Data Needs

학습 데이터셋 구성:

- **Examples**: 개별 데이터 포인트
- **Features**: 예시 내 속성
- **Labels**: 정답 또는 범주 정보

**데이터셋 사양 문서화:**

- 필요 데이터 유형과 형식
- 잠재적 데이터 소스
- 필요한 features와 labels
- 사용자 요구에 부합하는 성공 기준

**공정성 확보:**

ML 시스템은 다음을 통해 실패할 수 있다:

- **대표성 피해**: 부정적 고정관념 증폭
- **기회 거부**: 기회 접근을 제한하는 결정
- **불균형적 제품 실패**: 특정 사용자 그룹에 대한 치우친 출력
- **불이익에 의한 피해**: 인구통계와 행동 간 불리한 연관성 추론

### 3. Source Your Data Responsibly

**기존 데이터셋 평가 기준:**

- 사용자/사용 사례 적절성
- 수집 방법론
- 적용된 변환
- 필요한 보강
- 트레이드오프와 가정
- 준수 및 라이선스
- 문서화 (Data Cards)

**라이브 데이터 수집:**

- 암묵적 피드백: 백그라운드 사용자 활동 데이터
- 명시적 피드백: 직접적 사용자 응답

**PII 보호:**

- 개인 식별 정보(PII)와 보호 특성 검토
- 민감 데이터 수집 전 법률 전문가 상담
- 적대적 시나리오 대비
- 익명화 접근: 집계(aggregation), 편집(redaction)

### 4. Prepare and Document Your Data

**데이터셋 분할:**

- Training set: 모델 학습용
- Test set: 모델 성능 평가용 (미공개 데이터)
- 일반적 분할: 60% 학습, 40% 테스트

**Data Cards 문서화:**

- 데이터가 나타내는 것
- 데이터 외형
- 데이터 출처
- 준비 방법
- 사용 사례 적용성
- 책임있는 사용 가이드라인

### 5. Design for Labelers & Labeling

**라벨러 풀 다양성:**

- 주석자가 최종 사용자와 유사한지?
- 라벨러와 사용자 간 문화적 차이가 있는지?
- 가이드라인이 모국어로 제공되는지?

**라벨링 도구 설계:**

- 효율적 키 플로우를 위한 단축키 제공
- 마음 변경과 유연한 워크플로 허용
- 오류 자동 탐지 및 표시
- "확신 없음" 표시 또는 건너뛰기 허용
- 라벨러 간 불일치를 실세계 해석 차이에 대한 통찰로 가치화

### 6. Tune Your Model

**튜닝 접근:**

- What-If Tool, Language Interpretability Tool(LIT)로 블라인드스팟 검사
- 다양한 대상 사용자로 지속 테스트
- 사용자 피드백 메커니즘 구축
- 경험 모니터링용 커스텀 대시보드

**문제 해결 시:**

- 데이터 품질과 양 평가
- 의도하지 않은 결과 탐색: 출력 오류 패턴, 데이터 드리프트, 시스템 오류
- 특정 데이터 features, labels, 모델 파라미터로 문제 매핑

## Summary

Google PAIR Guidebook의 핵심은 AI 시스템 설계 시 **사용자 중심 접근**을 체계적으로 적용하는 것이다:

1. **User Needs**: AI 고유 가치가 있는 문제를 식별하고, 자동화 vs 증강을 결정
2. **Explainability + Trust**: 전략적 부분 설명으로 적절한 신뢰를 구축
3. **Mental Models**: 단계적 온보딩과 공동 학습으로 정확한 기대치 설정
4. **Feedback + Control**: 피드백을 모델 개선에 연결하고 사용자 제어와 자동화의 균형
5. **Data Collection**: 고품질 데이터를 처음부터 계획하고 책임있게 소싱

## 보강: Errors & Graceful Failure (심화)

PAIR 가이드북의 Mental Models, Feedback+Control 챕터에서 부분적으로 다루는 오류 처리를 심화 정리한다.

### 오류의 유형

AI 시스템의 오류는 크게 네 가지로 분류된다:

| 오류 유형 | 설명 | 예시 |
|---|---|---|
| **False Positive** | 없는 것을 있다고 판단 | 정상 이메일을 스팸으로 분류 |
| **False Negative** | 있는 것을 없다고 판단 | 악성 코드를 정상으로 통과 |
| **Out-of-scope** | 시스템 범위 밖의 요청 | 날씨 봇에 주식 정보 질문 |
| **Confidence failure** | 높은 확신으로 틀린 답 제공 | 잘못된 의료 진단을 99% 확신으로 제시 |

### 우아한 실패(Graceful Degradation) 설계 원칙

1. **실패를 인정한다**: "잘 모르겠다"를 말할 수 있는 시스템이 틀린 답을 확신하는 시스템보다 낫다
2. **대안을 제시한다**: AI가 실패하면 비AI 경로(수동 검색, 인간 상담원)로 안내한다
3. **실패에서 학습한다**: 오류 발생 시 사용자에게 피드백을 요청하고, 이를 모델 개선에 활용한다
4. **실패 비용을 비대칭적으로 설계한다**: false positive와 false negative의 비용이 다른 도메인에서는 더 안전한 쪽으로 편향시킨다

### 오류 발생 시 사용자 경험 설계

**즉각 대응:**

- 오류를 명확히 인지시킨다 ("요청을 처리할 수 없었다")
- 왜 실패했는지 사용자가 이해할 수 있는 수준으로 설명한다
- 다음 행동을 제안한다 ("다시 시도", "다르게 표현해보기", "직접 검색")

**복구 경로:**

- 실행 취소(undo) 기능을 항상 제공한다
- 수정이 쉬워야 한다 (한 번의 클릭/탭으로)
- 수정 결과가 즉시 반영되어야 한다
- 사용자의 수정이 향후 시스템 행동에 반영됨을 전달한다

**반복 오류 시:**

- 동일한 오류가 반복되면 다른 접근을 제안한다
- 시스템의 한계를 솔직히 전달하고 대안 경로를 강조한다
- 사용자가 좌절하기 전에 인간 에스컬레이션 경로를 제공한다

### 신뢰 회복 패턴

오류 후 신뢰 회복은 오류 방지보다 어렵다:

1. **투명한 인정**: 무엇이 잘못되었는지 숨기지 않는다
2. **원인 설명**: 가능하면 왜 오류가 발생했는지 설명한다
3. **재발 방지**: 같은 오류가 반복되지 않을 것임을 보여준다
4. **점진적 재신뢰**: 신뢰를 한 번에 회복하려 하지 말고 작은 성공을 축적한다

## 보강: Fairness & Inclusion (심화)

Data Collection 챕터에서 부분적으로 다루는 공정성을 심화 정리한다.

### AI 시스템의 공정성 차원

| 차원 | 설명 | 질문 |
|---|---|---|
| **대표성 (Representation)** | 학습 데이터가 실제 사용자 분포를 반영하는가 | 특정 인구집단이 과소/과대 대표되지 않는가? |
| **기회 균등 (Equal Opportunity)** | 같은 자격의 사용자가 같은 결과를 받는가 | 성별/인종에 따라 추천이 달라지지 않는가? |
| **결과 균등 (Equal Outcome)** | 결과의 분포가 집단 간 공정한가 | 특정 집단에 불이익이 집중되지 않는가? |
| **절차적 공정 (Procedural)** | 의사결정 과정이 투명하고 일관적인가 | 같은 입력에 같은 로직이 적용되는가? |

### 편향의 소스

편향은 ML 파이프라인의 모든 단계에서 유입된다:

1. **문제 정의 단계**: 어떤 문제를 풀 것인가의 선택 자체에 편향이 있을 수 있다
2. **데이터 수집**: 역사적 편향이 데이터에 반영 (예: 채용 데이터의 성별 편향)
3. **라벨링**: 라벨러의 주관과 문화적 맥락이 라벨에 반영
4. **모델 학습**: 다수 집단의 패턴에 과적합
5. **평가**: 평가 지표가 특정 집단에 유리하게 설계
6. **배포 후**: 피드백 루프가 기존 편향을 강화

### 공정성 실무 체크리스트

- [ ] 학습 데이터의 인구통계 분포를 분석했는가?
- [ ] 보호 속성(성별, 인종, 연령 등)별 모델 성능을 비교했는가?
- [ ] Facets, What-If Tool 등으로 데이터 편향을 시각화했는가?
- [ ] 모델이 보호 속성을 프록시로 사용하지 않는지 확인했는가?
- [ ] 다양한 사용자 그룹으로 사용자 테스트를 수행했는가?
- [ ] 배포 후 공정성 지표를 지속 모니터링하는 파이프라인이 있는가?
- [ ] 편향이 발견되었을 때의 대응 프로세스가 정의되어 있는가?

### 포용적 설계 원칙

1. **다양한 팀 구성**: 설계/개발/테스트 팀이 다양한 배경을 가져야 한다
2. **엣지 케이스 우선**: 주류가 아닌 사용자를 위한 설계가 모든 사용자에게 도움이 된다
3. **접근성**: 시각/청각/운동/인지 장애를 가진 사용자도 동등하게 사용할 수 있어야 한다
4. **문화적 맥락**: 날짜 형식, 이름 구조, 제스처 의미 등이 문화마다 다르다
5. **언어적 포용**: 비원어민, 방언 사용자, 비표준 표현도 지원한다

## 보강: Safety & Privacy (심화)

Data Collection 챕터에서 부분적으로 다루는 안전/개인정보를 심화 정리한다.

### AI 안전의 계층

| 계층 | 범위 | 예시 |
|---|---|---|
| **물리적 안전** | 시스템 오류가 신체적 위험을 초래 | 자율주행, 의료 진단, 산업 로봇 |
| **정보 안전** | 데이터 유출, 프라이버시 침해 | 개인정보 노출, 위치 추적 |
| **심리적 안전** | 사용자에게 정서적 피해 | 부적절한 콘텐츠 추천, 중독 유도 |
| **사회적 안전** | 집단적 피해, 사회적 영향 | 허위정보 확산, 선거 조작, 차별 강화 |

### 개인정보 보호 원칙

**데이터 최소화:**

- 목적 달성에 필요한 최소한의 데이터만 수집한다
- 수집 목적을 명확히 고지하고 동의를 받는다
- 목적 달성 후 데이터를 삭제하거나 익명화한다

**투명성:**

- 어떤 데이터가 수집되는지 사용자에게 명확히 전달한다
- 데이터가 어떻게 사용되는지 이해 가능한 언어로 설명한다
- 제3자 공유 여부를 명시한다

**사용자 제어:**

- 데이터 수집 옵트아웃 옵션 제공
- 수집된 데이터를 확인하고 삭제할 수 있는 경로
- 개인화 수준을 사용자가 조절할 수 있도록 허용
- 데이터 내보내기(portability) 지원

### AI 시스템의 안전 설계 패턴

**Human-in-the-Loop:**

- 고위험 결정에는 인간 검토를 필수화한다
- 자동화 수준을 위험 수준에 반비례하게 설정한다
- 긴급 상황에서 인간이 시스템을 즉시 중단할 수 있어야 한다

**Fail-Safe Default:**

- 시스템 실패 시 가장 안전한 상태로 복귀한다
- 불확실할 때는 행동하지 않는 쪽을 기본값으로 한다
- 중요 결정에서는 사용자 확인을 요구한다

**Adversarial Robustness:**

- 적대적 입력에 대한 방어 메커니즘 구축
- 입력 검증과 이상치 탐지
- 정기적 레드팀 테스트 수행

**모니터링 & 감사:**

- 배포 후 실시간 성능/안전 지표 모니터링
- 정기적 편향/공정성 감사
- 이상 탐지 시 자동 알림 및 에스컬레이션
- 감사 로그(audit trail) 유지
