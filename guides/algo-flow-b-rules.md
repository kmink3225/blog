# Flow B 문제 설계 원칙 + 실전 연착륙 규칙

> **언제 로드하는가**: Mode B(연습 문제 출제) 트리거가 감지될 때.
> **성공 기준**: Mode B Lv.N 3문제를 힌트 없이 통과하면, 동일 유형 실전 Lv.N 10문제 중 **7~8개를 예측 가능하게** 풀 수 있어야 한다.
> 이 파일의 모든 규칙은 Flow B의 모든 Step에 우선 적용된다.

---

## 바로 출제 트리거 (확인 단계 스킵)

사용자 입력에 아래 키워드가 있으면 모든 확인 단계를 건너뛰고 즉시 Step 1(문제 제시)로 진입한다.

| 트리거 | 효과 |
|--------|------|
| `go` 또는 `!` | 주제 선정 확인·앵커 노출 확인·기대 시간 확인·푸시 타이밍 확인을 **모두 스킵**. Self-Check·푸시는 세션 종료 시 일괄 |
| `random` | 주제 랜덤 선정을 즉시 확정 (선정 알림은 문제 상단에 1줄로 표기하되 사용자 응답 대기 없음) |
| 유형 직접 명시 (`Hash`, `DFS`, `문자열` 등) | 해당 유형으로 고정, 유형 확인 스킵 |

**트리거 조합 동작**

| 입력 예시 | 처리 |
|---|---|
| `/algo DS Lv1 문자열` | 확인 단계 유지 (기존 흐름) |
| `/algo DS Lv1 문자열 go` | 문자열 고정 + 즉시 출제 |
| `/algo DS Lv1 random go` | DS Lv.1 전체 유형 랜덤 + 즉시 출제 |
| `/algo AIE Lv3 DFS go` | AIE DFS 고정 + 즉시 출제 |

**주의**

- `go` 트리거가 있어도 Rule 10 자기 검증 8항목은 **내부적으로 반드시 수행**한다. 통과하지 못하면 출제를 중단하고 사용자에게 알린다 (스킵 대상은 "확인 질문"이지 "품질 검증"이 아니다)
- `go` 트리거는 Mode B·C에서만 유효. Mode A(실전 URL 제공)는 기본적으로 즉시 진행이므로 `go` 불필요

## 주제 선정 프로토콜

**모든 경우에 스크립트를 실행한다.** 사용자가 그룹을 지정했더라도 그룹 내 세부 토픽은 스크립트가 uniform random으로 선정한다.

```bash
# 트랙·레벨만 지정 → 그룹·토픽 모두 random
python scripts/tutoring_topic_picker.py --tutor algo --track DS --level N
python scripts/tutoring_topic_picker.py --tutor algo --track AIE --level N
python scripts/tutoring_topic_picker.py --tutor algo --track 공통 --level N

# 그룹 키워드 지정 → 그룹은 고정, 토픽만 random
python scripts/tutoring_topic_picker.py --tutor algo --track AIE --level 3 --group dfs
python scripts/tutoring_topic_picker.py --tutor algo --track DS --level 2 --group hash

# 그룹 번호 지정
python scripts/tutoring_topic_picker.py --tutor algo --track DS --level 1 --group 2

# seed 고정 (재현 필요 시)
python scripts/tutoring_topic_picker.py --tutor algo --track DS --level 2 --seed 42
```

JSON 출력에서 `track`, `group_name`, `topic` 필드를 추출한다.

**입력 → 스크립트 인수 매핑**

| 사용자 입력 | 스크립트 호출 |
|------------|--------------|
| `DS Lv1 random go` | `--track DS --level 1` (그룹·토픽 모두 random) |
| `AIE Lv3 DFS go` | `--track AIE --level 3 --group dfs` (그룹 고정, 토픽 random) |
| `DS Lv2 Hash go` | `--track DS --level 2 --group hash` |
| `AIE Lv3 백트래킹 go` | `--track AIE --level 3 --group backtrack` |
| `DS Lv4 DP go` | `--track DS --level 4 --group dp` |

**선정 결과 표기**

- `go` 트리거 **있음**: 즉시 출제, 선정 결과는 문제 상단 1줄로 표기
  ```
  ## 연습 문제: {제목}
  **난이도**: {track} Lv.N | **유형**: {group_name} — {topic} | **주제 선정**: {랜덤 / 그룹 지정}
  ```
- `go` 트리거 **없음**: 확인 대기 모드
  ```
  주제 선정: {track} Lv.N "[그룹 M: {group_name}] {topic}"
  다른 주제를 원하면 말해달라. 없으면 그대로 출제한다.
  ```

**그룹 키워드 목록** (`--group` 인수에 사용):

| 트랙 | 레벨 | 그룹명 | 키워드 예시 |
|------|------|--------|------------|
| DS | 1 | 데이터조작 | 데이터, 조작 |
| DS | 1 | 문자열처리 | 문자열, string |
| DS | 1 | Hash기초 | hash |
| DS | 2 | 정렬·비교 | 정렬, sort |
| DS | 2 | Hash심화 | hash |
| DS | 3 | 시뮬레이션 | 시뮬레이션, sim |
| DS | 3 | 통계·누적합 | 통계, prefix |
| DS | 4 | DP기초 | dp |
| DS | 4 | 이진탐색 | binary, 이진 |
| AIE | 1 | Array·Hash | array, hash |
| AIE | 2 | Stack·Queue | stack, queue |
| AIE | 2 | Two Pointer | pointer, sliding |
| AIE | 3 | DFS·BFS | dfs, bfs |
| AIE | 3 | 백트래킹 | backtrack |
| AIE | 4 | DP | dp |
| AIE | 4 | Tree·힙 | tree, heap |
| 공통 | 1 | 기본구현 | 구현, impl |
| 공통 | 3 | DFS·BFS | dfs, bfs |
| 공통 | 4 | 이진탐색 | binary, 이진 |

**주제 이력 추적**

- 같은 세션에서 출제한 topic을 기억한다 (최근 3개)
- 목적: 같은 주제가 연속으로 나오는 것 방지 → 스크립트의 uniform random과 시너지

## 문제 설계 원칙 (Flow B 전용)

**1. 함수 시그니처 정의**
- `def solution(입력매개변수) -> 반환타입:` 형태로 명확히 정의
- 입력 타입과 반환 타입을 명시한다

**2. 테스트 케이스 설계**
- 기본 케이스: 2~3개 (문제 이해 확인)
- Edge 케이스: 2~3개 (빈 배열, 단일 원소, 최대 크기, 음수, 중복 등)
- 총 5~6개 테스트 케이스를 제공한다

```python
# 테스트 케이스
assert solution([3,1,2,3]) == 2          # 기본
assert solution([1]) == 1                 # 단일 원소
assert solution([]) == 0                  # 빈 배열
assert solution([1,1,1,1]) == 1           # 모두 같은 값
assert solution(list(range(10000))) == 5000  # 최대 크기
```

**3. 제한사항 설계**
- 입력 크기 범위를 명시하여 기대 복잡도를 유도한다
- 예: "1 <= len(nums) <= 100,000" → O(n log n) 이하 필요

**4. 난이도 조절**
- Level 1~2: 단일 개념, 직관적 풀이
- Level 3: 단일 개념이지만 구현이 까다로움
- Level 4: 2개 개념 조합
- Level 5: 3개 이상 개념 조합 또는 비자명한 관찰 필요

**5. 트레이싱 검증**
- 사용자 코드를 테스트 케이스에 대해 수동 실행한다
- 각 단계의 변수 상태를 보여준다

```
예: "nums = [3,1,2,3], i=0 일 때:
  seen = {3}, result = 1
  i=1: seen = {3,1}, result = 2
  i=2: seen = {3,1,2}, result = 3
  i=3: 3은 이미 seen에 있으므로 skip
  → 최종 result = 3, max_pick = 2, 답 = min(3,2) = 2 ✓"
```

**6. 채점 신뢰도 (난이도별)**

| Level | 채점 방식 | 신뢰도 | 한계 고지 |
|-------|----------|--------|----------|
| 1~3 | 테스트 케이스 트레이싱 | 높음 | 불필요 |
| 4~5 | 테스트 케이스 트레이싱 | 중간 | 표기 |

- **Lv.1~3**: 테스트 케이스 수동 실행으로 충분히 정확한 채점이 가능하다. 별도 고지 없이 채점한다.
- **Lv.4~5**: 문제 제시 시 아래 문구를 포함한다:
  > Lv.4~5 연습 문제는 논리 검증으로 채점한다. 복잡한 상태 전이(DP), 재귀 종료 조건, 대규모 입력 edge case에서 오차가 있을 수 있다.

## 실전 연착륙 규칙 (Isomorphism Rules)

### Rule 1: Isomorphism 원칙 (최상위)

Mode B 문제는 **반드시 Programmers 기출 1개를 "앵커"로 선정**하고, 그 문제와 **구조적으로 동형**이 되도록 설계한다.

- **보존**: 사용 도구, 함정 카테고리, 기대 시간 복잡도, 독해 난이도
- **변경 허용**: 스토리, 변수명, 구체 수치
- **앵커 없는 창작 금지** — 앵커 기출을 특정할 수 없으면 출제하지 않는다

출제 시 문제 말미에 HTML 주석으로 앵커를 기록한다:

```
<!-- anchor: Programmers "{문제명}" (Lv.N, 정답률 XX%) -->
```

### Rule 2: 난이도별 정량 기준

| Level | 앵커 기출 정답률 | 최소 함정 수 | 공개/숨김 케이스 | 제약 상한 | 기대 풀이시간 |
|-------|----------------|-------------|----------------|----------|-------------|
| Lv.1  | 60~80%         | 2           | 3 / 10         | n ≤ 10⁴  | 5~15분      |
| Lv.2  | 40~60%         | 4           | 3 / 15         | n ≤ 10⁵  | 15~30분     |
| Lv.3  | 20~40%         | 6           | 4 / 20         | n ≤ 10⁶  | 30~60분     |
| Lv.4  | 10~20%         | 8           | 4 / 25         | n ≤ 10⁵  | 60~90분     |
| Lv.5  | < 10%          | 10          | 5 / 30         | n ≤ 10⁶  | 90분+       |

- 기준 미달이면 **문제로 채택하지 않는다**
- 함정 수 부족 → 함정 이식 후 재검토

### Rule 3: "쉬운 길" 차단 (하향 인플레이션 금지)

Mode B가 Mode A보다 쉬우면 연착륙 실패다. 다음 중 하나라도 해당하면 **부적합**:

- 단순 for loop + 조건문 하나로 풀림 (Lv.1조차 **관찰 또는 도구 선택**이 필요해야 함)
- 한 줄 내장 함수 호출로 끝남 (`return sorted(arr)` 수준)
- 브루트포스로도 제약 안에서 통과됨 → 제약 상한을 올려 브루트포스 차단
- 공개 예시만으로 모든 케이스가 커버됨 (함정이 실질 없음)

### Rule 4: 상향 인플레이션도 금지

- Lv.1인데 Lv.3 수준의 숨은 관찰 요구 금지
- Lv.2인데 비자명한 수학적 통찰 요구 금지
- 출제 전 **자기 검증 질문**: *"이 문제를 풀면 사용자가 앵커 기출의 정답률 XX%를 동일하게 경험할까?"*
- 자기 검증을 통과하지 못하면 난이도를 재조정한다

### Rule 5: 함정 이식 필수 (실전 예측의 핵심)

앵커 기출의 함정 카테고리를 **최소 60% 재현**한다.

**함정 카테고리 체크리스트** (매 문제 `최소 함정 수`개 이상 이식):

- [ ] 빈 입력 / 단일 원소 / 경계값 (0, 1, max)
- [ ] 중복 원소 / 동점 처리 / 정렬 안정성
- [ ] 대소문자 / 공백 / 특수문자 (문자열 유형)
- [ ] 음수 / 0 / 소수 / 정수 오버플로우
- [ ] off-by-one (인덱스 ±1, 반개구간)
- [ ] 입력 형식 혼합 (문자열 속 숫자, 2차원 배열 등)
- [ ] 최대 제약에서 시간 초과 (브루트포스 차단)
- [ ] 2차 정렬 키 / 복합 조건

### Rule 6: 공개/숨김 테스트 분리 (실전 채점 재현)

출제 시 테스트 케이스를 **두 세트로 분리**한다:

```
## 입출력 예 (공개)
- 기본 케이스 2~4개
- 함정 없음, 문제 이해 확인용

## 채점 테스트 케이스 (제출 후 공개)
- 함정 케이스를 여기 숨긴다
- 제출 시 하나씩 실행하며 결과 보고
```

**제출 후 피드백 형식** (Programmers 스타일 재현):

```
### 채점 결과
정확성 테스트
  테스트  1: 통과 (0.01ms)
  테스트  2: 통과 (0.01ms)
  테스트  3: 실패 — 기댓값 2, 실제 3
  테스트  4: 통과
  ...
  테스트 13: 실패 — 빈 문자열 처리 누락
  테스트 15: 통과

통과: 13 / 15
```

"예시만 맞고 제출 실패" 경험을 의도적으로 만든다 — 이것이 실전 함정 내성의 핵심 훈련이다.

### Rule 7: 실전 브리지 제시 (의무)

풀이 완료 후 **Step 6 (개념 정리) 다음에 반드시** 아래 블록을 포함한다:

```markdown
### 실전 연결 (Bridge to Mode A)

**앵커**: Programmers "{문제명}" (Lv.N, 정답률 XX%)
- 링크: https://school.programmers.co.kr/...

**동일 유형 추천 3문제**:
1. {문제A} (Lv.N) — 동일 도구, 약간 더 어려운 함정
2. {문제B} (Lv.N) — 동일 도구, 다른 스토리
3. {문제C} (Lv.N+1) — 도전용
```

이 블록이 없으면 Flow B 완료로 인정하지 않는다.

### Rule 8: 난이도 재조정 & Mode A 전환 트리거

| 상황 | 조치 |
|------|------|
| 1차 시도 **5분 이내** 통과 | 다음 문제: 함정 +1개 또는 Lv. +0.5 |
| 1차 시도 통과 (정상 시간) | 다음 문제: 동일 난이도, 다른 도구 조합 |
| **3회 이상** 실패 | 다음 문제: 동일 도구 복습 (쉬움이 아니라 각도만 다름) |
| 같은 유형·Lv. **3연속 1차 통과** | **Mode A 전환 강제 권고** — 실전 문제로 이동 |

### Rule 9: 출제 서술 톤 (실전 독해 훈련)

Mode B 문제의 서술은 **Programmers 기출 톤**을 모방한다.

| 항목 | Programmers 스타일 (따를 것) | 피할 것 |
|------|----------------------------|---------|
| 서두 | 스토리·맥락 ("어느 회사에서...") | "다음을 하시오" |
| 조건 배치 | 본문 속 분산 + 제한사항 박스 | 깔끔한 목록만 |
| 함수 시그니처 | `solution(s)`, `solution(arr, n)` 추상 변수 | `logs`, `target` 설명적 이름 |
| 지시문 | "~를 return 하도록 solution 함수를 완성하시오" | "~를 구하시오" |
| 입력 타입 | 때로 혼합 (문자열 속 숫자 등) | 항상 깔끔한 타입 |

독해 자체가 시험 대상이 되도록 **조건을 본문에 최소 1개 숨긴다** (제한사항 박스에도 중복 기재하므로 놓쳐도 복구 가능).

### Rule 10: 출제 전 자기 검증 체크리스트

출제 직전 아래 8항목을 **모두 통과**해야 한다. 하나라도 실패하면 재설계.
