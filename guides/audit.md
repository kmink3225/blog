# 콘텐츠 품질 감사 (Post-Completion Audit)

> **실행 시점**: 포스트 작성·수정·전환 등 콘텐츠 태스크가 완료된 직후, 최종 결과물에 대해 이 절차를 실행한다.

## 목적

작성된 콘텐츠에서 **독자가 한 번 읽고 이해하기 어려운 부분**을 식별하고 보강한다.
AGENT_GUIDE.md의 "주장-근거-해석 원칙"과 "추상 → 직관" 규칙을 사후 검증하는 체크리스트이다.

---

## Step 1: 감사 대상 수집

작업에서 생성·수정한 모든 `.qmd` 파일 목록을 확인한다.

---

## Step 2: 단락별 스캔

각 파일을 **섹션(## 단위) 순서로** 읽으며, 아래 7가지 감사 항목에 해당하는 단락을 표시한다.

### 감사 항목

| # | 항목 | 탐지 기준 | 예시 |
|---|------|----------|------|
| A | **정의만 있고 직관 없음** | 개념·용어를 정의했지만 "그래서 이게 뭔데?"에 답하는 비유·일상 대응이 없다 | "상위 온톨로지는 도메인 독립적 최상위 범주이다" → 독자: 그래서? |
| B | **분류·목록만 나열** | 표나 목록으로 항목을 나열했지만, 항목 간 차이가 체감되지 않는다. 각 항목이 실제로 어떤 상황에서 쓰이는지 시나리오가 없다 | 4단계 수준 체계 표만 있고, 각 수준이 실무에서 어떻게 만나지는지 설명이 없다 |
| C | **숫자·지표 해석 부재** | 숫자가 등장하지만 "so what"이 없다 | "정확도 0.92" → 이것이 좋은 건지 나쁜 건지 판단 기준이 없다 |
| D | **수식 직관 부재** | 수식 뒤에 "이 수식이 말하는 것은…"이 없다 | $R \subseteq C_1 \times C_2$ → 비전공자는 이것만으로 관계를 이해할 수 없다 |
| E | **주장 근거 부재** | "~해야 한다", "~가 중요하다" 뒤에 왜인지가 없다 | "재사용이 중요하다" → 왜? 안 하면 어떤 일이 생기는데? |
| F | **전문 용어 미해설** | 도메인 전문 용어가 처음 등장하는데 정의·풀이 없이 사용된다 | "OWA를 채택한다" → OWA가 뭔지 같은 글에서 설명한 적 없다 |
| G | **연결 고리 부재** | 섹션 간, 또는 개념 간 "왜 이 순서인가", "앞 내용과 어떻게 이어지는가"가 없다 | 설계 방법론 → 설계 패턴으로 넘어갈 때 전환 맥락이 없다 |

---

## Step 3: 보강 방식 판단

감사에서 발견된 각 항목에 대해 아래 기준으로 보강 방식을 결정한다.

### 판단 기준표

| 조건 | 보강 방식 | 설명 |
|------|----------|------|
| 1~3문장 추가로 해소 가능 | **인라인 보강** | 해당 단락 바로 아래에 직관적 설명·비유·반사실 시나리오를 추가한다 |
| 별도 예시 블록(callout, 표, 코드)이 필요하지만 해당 섹션 범위 내 | **섹션 내 확장** | 해당 `##`/`###` 섹션 안에 하위 섹션 또는 callout 블록을 추가한다 |
| 설명이 길어져서(500자 이상) 해당 포스트의 흐름을 깨뜨릴 때 | **별도 포스트 분리** | 새 포스트를 만들고, 원본에서 크로스 링크를 건다. 원본에는 1~2문장 요약 + 링크만 남긴다 |
| 여러 포스트에서 반복적으로 필요한 배경 지식일 때 | **별도 포스트 분리** | 공통 배경 지식 포스트를 만들고, 필요한 모든 포스트에서 링크한다 |

### 별도 포스트 분리 시 절차

별도 포스트가 필요하다고 판단되면, `guides/write-post.md`의 Step 1~6을 그대로 따른다.

1. 카테고리 확인 + GUIDE.md 읽기
2. 기존 콘텐츠 중복 판단 (이미 설명하는 포스트가 있는지 반드시 확인)
3. 포스트 작성 (AGENT_GUIDE.md 작성 규칙 준수)
4. index.qmd 업데이트
5. 원본 포스트에 크로스 링크 추가
6. 관련 카테고리의 index.qmd에도 크로스 링크 추가 (해당 시)

---

## Step 4: 보강 실행

### 인라인 보강 패턴 (항목별 1줄 요약)

| 항목 | 보강 방법 |
|------|----------|
| A. 정의만 있고 직관 없음 | 정의 직후에 일상 비유, 반사실("X가 없다면?"), 또는 구체 시나리오를 추가 |
| B. 분류·목록만 나열 | 표 아래에 하나의 연속 시나리오로 각 항목을 관통하는 예시 추가 |
| C. 숫자·지표 해석 부재 | 숫자 뒤에 "실무에서 이것이 뜻하는 바" 1~2문장 추가 |
| D. 수식 직관 부재 | 수식 직후에 "이 수식이 말하는 것은" 1~2문장 추가 |
| E. 주장 근거 부재 | "~해야 한다" 뒤에 "그렇지 않으면 ~가 발생한다" (반사실) 추가 |
| F. 전문 용어 미해설 | 처음 등장 위치에 괄호 풀이 또는 각주 추가 |
| G. 연결 고리 부재 | 섹션 시작부에 전환 문장 추가 ("앞의 X를 기반으로, 여기서는 Y를 다룬다") |

---

## Step 5: 보강 후 재검증

보강이 완료되면 다음을 확인한다.

| 검증 항목 | 확인 내용 |
|----------|----------|
| 스타일 일관성 | 보강한 부분이 한다 체를 유지하는가 |
| 헤더 번호 금지 | 추가한 하위 섹션에 수동 번호를 붙이지 않았는가 |
| 줄 개행 | 보강한 본문 줄 끝에 공백 2칸이 있는가 |
| LaTeX 공백 | 추가한 수식의 `$` 양쪽에 공백이 있는가 |
| 크로스 링크 정합성 | 별도 포스트를 만들었다면 양방향 링크가 모두 걸려 있는가 |
| index.qmd | 별도 포스트를 만들었다면 index.qmd에 추가했는가 |
| 과잉 보강 방지 | 원래 글의 흐름을 깨뜨리지 않았는가. 보강이 본문보다 길어지면 별도 포스트 분리를 재검토한다 |

---

## Step 6: 포스트 분할 적정성 검증 (convert-tbd 태스크 시 필수)

`convert-tbd` 태스크로 생성된 포스트에 대해, 분할 판단이 올바르게 수행되었는지 사후 검증한다.

### 검증 항목

| # | 항목 | 탐지 기준 | 조치 |
|---|------|----------|------|
| H | **과대 포스트** | 단일 포스트가 200줄을 초과하고, 내부에 독립 완결 가능한 주제가 2개 이상 존재한다 | 분할 필요 — `convert-tbd.md` Step 2 기준표를 다시 적용하여 분할한다 |
| I | **독자 관심사 혼재** | 하나의 포스트 안에 서로 다른 배경 지식을 전제하는 섹션이 공존한다. 예: 프롬프트 설계 방법론 + ERD 비교 → 전자는 AI 실무자, 후자는 SW 엔지니어가 관심 대상 | 분할 검토 — 독자별로 포스트를 나눌 수 있는지 판단한다 |
| J | **과소 분할** | 분할된 포스트가 50줄 미만으로 너무 짧아 독립 포스트로서 가치가 부족하다 | 인접 포스트와 병합을 검토한다 |

### 검증 절차

1. 생성된 각 포스트의 **줄 수**를 확인한다
2. 200줄 초과 포스트가 있으면, 내부 `##` 섹션을 스캔하여 **독립 완결 가능한 주제 덩어리**가 몇 개인지 센다
3. 2개 이상이면 **분할이 누락된 것**이다 — `convert-tbd.md` Step 2를 재실행한다
4. 50줄 미만 포스트가 있으면 **과소 분할**을 검토한다

::: {.callout-important}
## 이 검증이 필요한 이유

`convert-tbd.md` Step 2의 분할 판단을 agent가 형식적으로만 수행하고 실제 기준 대입을 건너뛰는 사례가 발생했다. 이 검증은 분할 판단의 사후 안전망이다. convert-tbd 태스크 완료 후에는 **반드시** 이 항목(H, I, J)을 확인한다.
:::

## 감사 실행 요약

```
태스크 완료
    ↓
Step 1: 감사 대상 파일 목록 수집
    ↓
Step 2: 섹션별 스캔 → 감사 항목 A~G 탐지
    ↓
Step 3: 항목별 보강 방식 판단 (인라인 / 섹션 확장 / 별도 포스트)
    ↓
Step 4: 보강 실행
    ↓
Step 5: 재검증 (스타일, 링크, 과잉 보강)
    ↓
Step 6: 포스트 분할 적정성 검증 (convert-tbd 시 필수, 감사 항목 H~J)
    ↓
완료
```
