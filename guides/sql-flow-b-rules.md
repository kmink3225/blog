# Flow B 문제 설계 원칙 + 실전 연착륙 규칙 (SQL)

> **언제 로드하는가**: Mode B(연습 문제 출제) 트리거가 감지될 때.
> **성공 기준**: Mode B Lv.N 3문제를 힌트 없이 통과하면, 동일 유형 실전 Lv.N 10문제 중 **7~8개를 예측 가능하게** 풀 수 있어야 한다.
> 이 파일의 모든 규칙은 Flow B의 모든 Step에 우선 적용된다.

---

## 바로 출제 트리거 (확인 단계 스킵)

사용자 입력에 아래 키워드가 있으면 모든 확인 단계를 건너뛰고 즉시 Step 1(문제 제시)로 진입한다.

| 트리거 | 효과 |
|--------|------|
| `go` 또는 `!` | 주제 선정 확인·앵커 노출 확인·기대 시간 확인·푸시 타이밍 확인을 **모두 스킵**. Self-Check·푸시는 세션 종료 시 일괄 |
| `random` | 주제 랜덤 선정을 즉시 확정 (선정 알림은 문제 상단에 1줄로 표기하되 사용자 응답 대기 없음) |
| 유형 직접 명시 (`JOIN`, `NULL처리` 등) | 해당 유형으로 고정, 유형 확인 스킵 |

**트리거 조합 동작**

| 입력 예시 | 처리 |
|---|---|
| `/sql Lv2 JOIN` | 확인 단계 유지 (기존 흐름) |
| `/sql Lv2 JOIN go` | JOIN 고정 + 즉시 출제 |
| `/sql DS Lv1 random go` | DS Lv.1 전체 유형 랜덤 + 즉시 출제 |
| `/sql Lv3 random` | 랜덤 선정 즉시 확정, 다만 앵커·기대 시간 안내는 문제와 함께 표기 |

**주의**

- `go` 트리거가 있어도 Rule 10 자기 검증 8항목은 **내부적으로 반드시 수행**한다. 통과하지 못하면 출제를 중단하고 사용자에게 알린다 (스킵 대상은 "확인 질문"이지 "품질 검증"이 아니다)
- `go` 트리거는 Mode B·C에서만 유효. Mode A(실전 URL 제공)는 기본적으로 즉시 진행이므로 `go` 불필요

## 주제 선정 규칙 (랜덤 선정)

사용자가 **구체적 주제(유형·개념)를 지정하지 않으면** 아래 절차로 랜덤 선정한다.

**"구체적 주제 지정" 판정 기준**

| 지정됨 (사용자 의도 명확) | 미지정 (랜덤 선정 대상) |
|---|---|
| "GROUP BY 문제", "JOIN 연습" | "Lv.1 연습", "DS 문제" |
| "NULL 처리", "HAVING 문제" | "sql 문제 내줘", "Lv.2" |
| "Window Function Lv.4" | "문제 만들어줘" |

**랜덤 선정 절차**

1. 지정된 Level의 공식 하위 유형 목록에서 후보 추출  
   (예: Lv.1 → SELECT/WHERE, ORDER BY/LIMIT, DISTINCT/NULL, 기본 집계, 혼합)
2. 이전 3문제에서 나온 유형은 **제외** (다양성 보장)
3. 남은 후보 중 랜덤 1개 선택
4. **선정 결과를 사용자에게 알린다**:

   - `go` 트리거 **없음**: 확인 대기 모드
     ```
     주제 랜덤 선정: Lv.1 "기본 집계 (COUNT/SUM/AVG)"
     다른 주제를 원하면 말해달라. 없으면 그대로 출제한다.
     ```
   - `go` 트리거 **있음**: 즉시 출제, 선정 결과는 문제 상단에 1줄로 표기
     ```
     ## 연습 문제: {제목}
     **난이도**: Lv.1 | **핵심 기능**: 기본 집계 | **주제 선정**: 랜덤 (go trigger)
     ```

5. `go` 없는 경우 사용자가 별도 지시 없이 답하면 그대로 출제 진행
6. 사용자가 "다른 걸로" 하면 후보에서 해당 유형 제외 후 재선정

**주제 이력 추적**

- 같은 세션에서 출제한 주제를 기억한다 (최근 3개)
- 목적: 같은 유형이 연속으로 나오는 것 방지 → 실전 대비 커버리지 확보

## 문제 설계 원칙 (Flow B 전용)

**1. 테이블 설계**

- 테이블 수: Lv.1 = 1개, Lv.2~3 = 2~3개, Lv.4~5 = 3개 이상
- 컬럼 수: 테이블당 3~6개
- 테이블/컬럼명: 영어, 현실적 이름 (`employees`, `orders`, `products`) — `tbl_a` 금지

**2. 샘플 데이터 (공개)**

- 행 수: Lv.1=6, Lv.2=6, Lv.3=8, Lv.4=10, Lv.5=12 (손 트레이싱 가능한 크기)
- 기본 케이스만 포함, 함정은 **포함하지 않음** (공개 예시에서 함정이 드러나면 실전감 소실)
- INSERT 문으로 제공

**3. 채점 데이터 (숨김)**

- 행 수: 공개 샘플과 동일 수준
- **함정 케이스를 반드시 포함**: NULL, 중복, 경계값, 빈 결과
- 사용자 제출 후에만 공개

**4. 기대 결과**

- 공개 샘플 기준 정확한 결과 테이블 제시
- 정렬 순서가 중요하면 명시
- 동점 처리 규칙이 있으면 문제에 서술 (또는 의도적으로 모호하게 두어 함정으로 활용)

**5. 트레이싱 검증 방법 (채점)**

- 사용자 쿼리를 샘플·채점 데이터에 대해 수동 트레이싱
- 각 절을 순서대로 적용: `FROM → WHERE → GROUP BY → HAVING → SELECT → ORDER BY → LIMIT`
- 중간 결과 테이블을 보여주며 어디서 틀렸는지 시각적으로 설명

```
예: "FROM employees e JOIN departments d ON e.dept_id = d.id 결과:
| e.name | d.name | e.salary |
|--------|--------|----------|
| Kim    | Dev    | 5000     |
| Lee    | Sales  | 3000     |
→ 여기까지는 맞다. WHERE 절에서 salary > 4000을 적용하면..."
```

**6. 채점 신뢰도 (난이도별)**

| Level | 채점 방식 | 신뢰도 | 한계 고지 |
|-------|----------|--------|----------|
| 1~3 | 논리 트레이싱 | 높음 | 불필요 |
| 4~5 | 논리 트레이싱 | 중간 | 표기 |

- **Lv.1~3**: 샘플 데이터 수동 실행으로 충분히 정확한 채점이 가능하다. 별도 고지 없이 채점한다.
- **Lv.4~5**: 문제 제시 시 아래 문구를 포함한다:
  > Lv.4~5 연습 문제는 논리 검증으로 채점한다. Window Function 프레임 경계, 재귀 CTE 종료 조건 등에서 edge case 오차가 있을 수 있다.

## 실전 연착륙 규칙 (Isomorphism Rules)

### Rule 1: Isomorphism 원칙 (최상위)

Mode B 문제는 **반드시 Programmers SQL 기출 1개를 "앵커"로 선정**하고, 그 문제와 **구조적으로 동형**이 되도록 설계한다.

- **보존**: 사용 SQL 기능, 함정 카테고리, 테이블 구조 패턴, 기대 쿼리 복잡도
- **변경 허용**: 스토리(도메인), 테이블·컬럼 이름, 샘플 데이터 값
- **앵커 없는 창작 금지** — 앵커 기출을 특정할 수 없으면 출제하지 않는다

출제 시 문제 말미에 HTML 주석으로 앵커를 기록한다:

```
<!-- anchor: Programmers "{문제명}" (SQL Lv.N, 정답률 XX%) -->
```

### Rule 2: 난이도별 정량 기준

| Level | 앵커 정답률 | 최소 함정 수 | 공개/숨김 행 수 | 테이블 수 | 기대 풀이시간 |
|-------|-----------|-------------|----------------|----------|-------------|
| Lv.1  | 70~90%    | 1           | 6 / 4 (총 10행)  | 1        | 3~8분       |
| Lv.2  | 50~70%    | 2           | 6 / 6 (총 12행)  | 2        | 8~15분      |
| Lv.3  | 30~50%    | 3           | 8 / 8 (총 16행)  | 2~3      | 15~25분     |
| Lv.4  | 15~30%    | 4           | 10 / 10 (총 20행)| 3        | 25~40분     |
| Lv.5  | < 15%     | 5           | 12 / 13 (총 25행)| 3+       | 40분+       |

- 기준 미달이면 **문제로 채택하지 않는다**
- 함정 수 부족 → 함정 이식 후 재검토
- 알고리즘과 달리 SQL은 입력 크기(n)가 아니라 **테이블 수·함정 수**로 난이도를 조절한다

### Rule 3: "쉬운 길" 차단 (하향 인플레이션 금지)

Mode B가 Mode A보다 쉬우면 연착륙 실패다. 다음 중 하나라도 해당하면 **부적합**:

- 한 테이블 단순 `SELECT * FROM t WHERE col = 값` 하나로 풀림 (Lv.1조차 최소 1개 관찰 필요: `IS NULL`, `LIKE`, `IN`, `BETWEEN`, 또는 정렬·집계)
- 공개 샘플만으로 모든 케이스 커버됨 (숨김 함정이 실질 없음)
- 기대 결과가 샘플 데이터 그대로 (필터·변환·집계 없음)
- Programmers 동일 Lv. 정답률 90%+ 문제와 완전 동형이면 부적합 (너무 쉬움)

### Rule 4: 상향 인플레이션도 금지

- **Lv.1에 금지**: Window Function, CTE, 상관 서브쿼리, JOIN
- **Lv.2에 금지**: 재귀 CTE, `ROWS BETWEEN` 복합 프레임
- **Lv.3에 금지**: Island & Gaps, PIVOT
- 출제 전 **자기 검증 질문**: *"이 문제를 풀면 사용자가 앵커 기출의 정답률 XX%를 동일하게 경험할까?"*
- 자기 검증을 통과하지 못하면 난이도를 재조정한다

### Rule 5: 함정 이식 필수 (SQL 특화)

앵커 기출의 함정 카테고리를 **최소 60% 재현**한다.

**SQL 함정 카테고리 체크리스트** (매 문제 `최소 함정 수`개 이상 이식):

- [ ] NULL 값 처리 — 집계에서 NULL 무시, JOIN에서 NULL 매칭, `WHERE col = NULL` 오류
- [ ] GROUP BY 누락 컬럼 — SELECT에 집계 없는 컬럼을 넣으면 오류/비결정적
- [ ] JOIN 조건 누락 — 카티션 곱 발생
- [ ] INNER vs LEFT JOIN — 매칭 없는 행 제외 vs 포함
- [ ] WHERE vs HAVING — 집계 전/후 필터링 혼동
- [ ] DISTINCT 필요 여부 — 중복 제거 누락 또는 불필요 남용
- [ ] 날짜 함수 방언 — `DATE_FORMAT`, `TO_CHAR`, `STRFTIME` 등
- [ ] 빈 결과 집계 — `COUNT`는 0, `SUM/AVG`는 NULL
- [ ] 정렬 기준 모호 — 동점 시 2차 정렬 키 누락
- [ ] 대소문자 / 공백 — 문자열 비교, `LIKE` 와일드카드
- [ ] 0 / 음수 / 빈 문자열 / 경계값
- [ ] 중복 행 가정 오류 — 고유성 없는 컬럼 기준 조인

### Rule 6: 공개/숨김 데이터 분리 (실전 채점 재현)

출제 시 샘플 데이터를 **두 세트로 분리**한다:

```
## 샘플 데이터 (공개)
- 기본 케이스만 (Lv별 공개 행 수)
- 함정 없음, 문제 이해 확인용
- INSERT 문으로 제시

## 채점 데이터 (제출 후 공개)
- 함정 케이스를 여기 숨긴다 (NULL, 중복, 경계값, 빈 결과 등)
- 제출 시 하나씩 실행하며 결과 보고
```

**제출 후 피드백 형식** (Programmers 스타일 재현):

```
### 채점 결과
정확성 테스트
  케이스 1 (공개): 통과
  케이스 2 (공개): 통과
  케이스 3 (공개): 통과
  케이스 4 (숨김, NULL 포함): 실패 — 기댓값 3건, 실제 2건
  케이스 5 (숨김, 중복 값): 통과
  케이스 6 (숨김, 빈 결과 집계): 실패 — 기댓값 0, 실제 NULL
  케이스 7 (숨김, 경계값): 통과

통과: 5 / 7
```

"공개 샘플은 다 맞는데 숨김에서 절반 틀림" 경험을 의도적으로 만든다 — 실전 SQL 함정 내성의 핵심 훈련이다.

### Rule 7: 실전 브리지 제시 (의무)

풀이 완료 후 **Step 6 (개념 정리) 다음에 반드시** 아래 블록을 포함한다:

```markdown
### 실전 연결 (Bridge to Mode A)

**앵커**: Programmers "{문제명}" (SQL Lv.N, 정답률 XX%)
- 링크: https://school.programmers.co.kr/learn/courses/30/parts/...

**동일 유형 추천 3문제**:
1. {문제A} (Lv.N) — 동일 기능, 약간 더 어려운 함정
2. {문제B} (Lv.N) — 동일 기능, 다른 도메인
3. {문제C} (Lv.N+1) — 도전용
```

이 블록이 없으면 Flow B 완료로 인정하지 않는다.

### Rule 8: 난이도 재조정 & Mode A 전환 트리거

| 상황 | 조치 |
|------|------|
| 1차 시도 **기대 풀이시간 절반 이내** 통과 | 다음 문제: 함정 +1개 또는 Lv. +0.5 |
| 1차 시도 통과 (정상 시간) | 다음 문제: 동일 난이도, 다른 기능 조합 |
| **3회 이상** 실패 | 다음 문제: 동일 기능 복습 (쉬움이 아니라 각도만 다름) |
| 같은 유형·Lv. **3연속 1차 통과** | **Mode A 전환 강제 권고** — 실전 문제로 이동 |

### Rule 9: 출제 서술 톤 (실전 독해 훈련)

Mode B 문제의 서술은 **Programmers 기출 톤**을 모방한다.

| 항목 | Programmers 스타일 (따를 것) | 피할 것 |
|------|----------------------------|---------|
| 서두 | 스토리·맥락 ("어느 쇼핑몰의 주문 데이터에서...") | "다음을 하시오" |
| 테이블 이름 | `orders`, `customers`, `products` 현실적 | `tbl_a`, `t1`, `test_table` |
| 조건 배치 | 본문 속 분산 + 기대 결과 | 깔끔한 목록만 |
| 지시문 | "~하는 쿼리를 작성하시오" | "~를 구하시오" |
| 함정 힌트 | 명시하지 않음 (본문에 조건만 깔아두고 놓치면 틀림) | "NULL을 주의하세요" 같은 힌트 |

독해 자체가 시험 대상이 되도록 **조건을 본문에 최소 1개 숨긴다** (기대 결과 테이블로 역추론 가능하므로 복구 여지는 있음).

### Rule 10: 출제 전 자기 검증 체크리스트

출제 직전 아래 8항목을 **모두 통과**해야 한다. 하나라도 실패하면 재설계.

- [ ] 주제 랜덤 선정이면 사용자에게 알렸는가? (Rule 0)
- [ ] 앵커 기출을 특정했는가? (이름·난이도·정답률 기록)
- [ ] 앵커 함정 카테고리를 60% 이상 이식했는가?
