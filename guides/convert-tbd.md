# 임시 메모 → 블로그 전환

사용자가 AI agent와의 대화에서 유용한 정보를 `docs/blog/posts/TBD.qmd`에 저장해 둘 수 있다. Agent는 이를 주제별 정식 블로그 포스트로 전환한다.

> **선행 조건**: 이 스킬을 실행하기 전에 `guides/info-search.md`의 병렬 탐색 프로토콜을 먼저 실행한다.

## 전환 절차

### Step 1: 임시 파일 읽기

내용을 파악하고 주제별로 분리한다.

### Step 2: 분할 판단 (필수 출력)

하나의 TBD 내용이 반드시 1개 포스트가 될 필요는 없다. 아래 기준표의 **4개 항목 모두**에 대해 판단 결과를 명시적으로 출력한 후 다음 Step으로 넘어간다. 판단 결과를 출력하지 않고 다음 Step으로 넘어가면 안 된다.

| 기준 | 1개 포스트 | 복수 포스트 분할 |
|------|----------|---------------|
| 주제 독립성 | 하나의 논점으로 수렴 | 각 주제가 독립적으로 완결 가능 |
| 예상 분량 | 합쳐도 적정 길이 (200줄 이하) | 합치면 지나치게 길어짐 (200줄 초과) |
| 독자 관점 | 하나의 글로 읽는 것이 자연스러움 | 관심사가 다른 독자가 각각 찾아 읽을 수 있음 |
| 시의성 | 모든 내용의 유효 기간이 비슷 | 일부는 장기 유효, 일부는 빠르게 outdated |

::: {.callout-important}
## Step 2 출력 형식 (반드시 이 형식으로 출력한다)

```
## 분할 판단 결과

| 기준 | 판단 | 근거 |
|------|------|------|
| 주제 독립성 | 분할 / 통합 | (구체적 이유) |
| 예상 분량 | 분할 / 통합 | (예상 줄 수 또는 분량 근거) |
| 독자 관점 | 분할 / 통합 | (대상 독자 차이 또는 동일성) |
| 시의성 | 분할 / 통합 | (유효 기간 차이 여부) |

**결론**: N개 포스트로 분할한다 / 1개 포스트로 통합한다
**분할 시 포스트 구성**:
1. (포스트 1 제목 — 포함 내용 요약)
2. (포스트 2 제목 — 포함 내용 요약)
...
```

4개 항목 중 **2개 이상이 "분할"이면 복수 포스트로 분할한다.**
판단이 애매할 때는 분할 쪽을 선택한다 — 나중에 합치는 것보다 처음부터 나누는 것이 쉽다.
:::

분할 시 포스트 간 크로스 링크를 반드시 추가한다. 시의성이 높아 빠르게 outdated되는 내용(가격, rate limit, 특정 모델 버전 비교 등)은 포스트로 전환하지 않는다.

### Step 3: 중복 확인 및 통합 판단 (필수)

이미 정식 포스트로 작성된 내용이 있는지 대조한다.

**확인 방법**:

1. 해당 카테고리 폴더의 파일 목록을 확인한다
2. 핵심 키워드로 `Grep`을 사용해 기존 파일을 검색한다
3. 후보 파일의 YAML 헤더 + 마크다운 헤더만 먼저 읽어 범위를 파악한다
4. 겹침이 의심되는 파일만 선택적으로 본문을 읽는다

**판단 기준 — 새 포스트 생성 vs 기존 포스트 통합**:

| 상황 | 조치 |
|------|------|
| 기존 포스트가 TBD 내용의 대부분을 이미 커버 | 새 파일 생성 금지 — 기존 파일을 면밀히 검토 후 누락된 내용만 추가 |
| 기존 포스트가 존재하지만 얕거나 관점이 다름 | 기존 포스트에 섹션을 추가하거나, 새 파일로 분리 후 상호 링크 |
| 유사 주제이지만 TBD 내용이 독립적으로 완결 가능 | 새 파일 작성, 기존 파일에 크로스 링크 추가 |
| 해당 주제 포스트가 전혀 없다 | 새 파일 작성 |

::: {.callout-important}
## 통합 우선 원칙

TBD 내용과 기존 포스트가 주제적으로 겹친다고 판단한 경우, **기존 포스트 전체를 먼저 읽고** TBD 내용과 정밀 비교한다.
"중복"이라고 판단해도 고유한 관점, 구체적 사례, 날카로운 프레이밍이 남아있을 수 있다.
그 고유 내용을 기존 포스트에 통합하는 것이 새 포스트를 만드는 것보다 우선이다.
중복된 내용을 기존 포스트에 추가할 때는 이미 있는 설명과 중복되지 않도록 주의한다.
:::

**기존 포스트 보강 시 주의사항**:

- 기존 파일의 전체 내용을 읽고 구조를 파악한 후에만 수정한다
- 빠진 내용·관점·예시만 추가한다 — 기존 내용을 삭제하거나 축소하지 않는다
- 새로 추가한 내용이 기존 내용과 중복되지 않는지 최종 확인한다

### Step 4: 분류

1. **맥락 도메인 나열**: 원본 메모가 어떤 분야의 대화에서 나온 내용인지 모두 나열한다 (예: Agent 성능 개선 맥락 → Agent, AI)
2. **카테고리 매핑**: 각 도메인을 블로그 카테고리에 매핑한다
3. **물리적 위치 결정**: 주 카테고리 폴더를 정한다 (핵심 원리가 속한 분야 기준)
4. **YAML categories 작성**: 주 카테고리 + 맥락 카테고리를 **모두** 포함한다
5. **크로스 링크 계획**: 관련 카테고리의 index.qmd에 크로스 링크를 추가할 대상을 미리 정한다

::: {.callout-important}
물리적 위치(주 카테고리)만 태깅하고 원본 대화의 맥락 카테고리를 누락하는 실수가 반복된다.
Step 4를 완료하기 전에 "원본 메모의 대화 맥락에서 파생되는 카테고리를 빠뜨리지 않았는가?"를 반드시 점검한다.
:::

### Step 5: 재작성 및 보강

- 대화 요소 제거 ("당신", "네", 독자에게 던지는 질문 등)
- 한다 체로 통일, YAML 헤더 포함
- **단순 스타일 변환이 아니라** 배경 설명, 수식, 코드, 다이어그램 등을 보강
- 독자가 메모의 맥락을 모르더라도 포스트만으로 완결되도록 작성

::: {.callout-important}
## 재작성 시 가장 빈번한 실수 2가지

**1. 원본의 구체적 예시를 추상적 한 줄로 축소한다.**
원본에 코드 블록, before/after 비교, 구체적 시나리오가 있으면 포스트에 **반드시 포함**한다. "~와 동일하다"라는 한 줄 요약으로 대체하지 않는다. 구체적 예시가 빠진 원칙은 독자에게 전달되지 않는다.

**2. 섹션을 나열만 하고 섹션 간 연결("왜 이 순서인가")을 생략한다.**
각 섹션 시작 시 "앞에서 X를 했으므로, 이제 Y를 한다"는 전환 문장을 넣는다.
:::

### Step 6: index.qmd 업데이트

- 주 카테고리의 index.qmd에 새 포스트 링크를 추가한다
- Step 4에서 정한 **관련 카테고리의 index.qmd에도 크로스 링크를 추가**한다

### Step 7: TBD.qmd 초기화

전환 완료 후 빈 템플릿 상태로 되돌린다 (파일 삭제 금지).

```markdown
---
title: "TBD"
description: |
  임시 메모 보관. 주제별 정식 블로그 포스트로 전환 후 내용을 비운다.
draft: true
---
```

## 임시 파일 규칙

- 경로: `docs/blog/posts/TBD.qmd` (고정)
- 하나의 파일에 여러 주제를 누적하며, 주제 간 구분은 `---` 또는 `## 제목`
- YAML 헤더에 `draft: true`를 포함하여 빌드에서 제외
- 전환 후에도 파일은 삭제하지 않고 빈 상태로 유지
