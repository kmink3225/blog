# 임시 메모 → 블로그 전환

사용자가 AI agent와의 대화에서 유용한 정보를 `docs/blog/posts/TBD.qmd`에 저장해 둘 수 있다. Agent는 이를 주제별 정식 블로그 포스트로 전환한다.

> **선행 조건**: 이 스킬을 실행하기 전에 `guides/info-search.md`의 병렬 탐색 프로토콜을 먼저 실행한다.

## 전환 절차

### Step 1: 임시 파일 읽기

내용을 파악하고 주제별로 분리한다.

### Step 2: 분할 판단

하나의 TBD 내용이 반드시 1개 포스트가 될 필요는 없다. 아래 기준으로 복수 포스트 분할 여부를 판단한다:

| 기준 | 1개 포스트 | 복수 포스트 분할 |
|------|----------|---------------|
| 주제 독립성 | 하나의 논점으로 수렴 | 각 주제가 독립적으로 완결 가능 |
| 예상 분량 | 합쳐도 적정 길이 | 합치면 지나치게 길어짐 |
| 독자 관점 | 하나의 글로 읽는 것이 자연스러움 | 관심사가 다른 독자가 각각 찾아 읽을 수 있음 |
| 시의성 | 모든 내용의 유효 기간이 비슷 | 일부는 장기 유효, 일부는 빠르게 outdated |

분할 시 포스트 간 크로스 링크를 반드시 추가한다. 시의성이 높아 빠르게 outdated되는 내용(가격, rate limit, 특정 모델 버전 비교 등)은 포스트로 전환하지 않는다.

### Step 3: 중복 확인

이미 정식 포스트로 작성된 내용이 있는지 대조한다.

### Step 4: 분류

1. **맥락 도메인 나열**: 원본 메모가 어떤 분야의 대화에서 나온 내용인지 모두 나열한다 (예: Agent 성능 개선 맥락 → Agent, AI)
2. **카테고리 매핑**: 각 도메인을 블로그 카테고리에 매핑한다
3. **물리적 위치 결정**: 주 카테고리 폴더를 정한다 (핵심 원리가 속한 분야 기준)
4. **YAML categories 작성**: 주 카테고리 + 맥락 카테고리를 **모두** 포함한다
5. **크로스 링크 계획**: 관련 카테고리의 index.qmd에 크로스 링크를 추가할 대상을 미리 정한다

::: {.callout-important}
물리적 위치(주 카테고리)만 태깅하고 원본 대화의 맥락 카테고리를 누락하는 실수가 반복된다.
Step 4를 완료하기 전에 "원본 메모의 대화 맥락에서 파생되는 카테고리를 빠뜨리지 않았는가?"를 반드시 점검한다.
:::

### Step 5: 재작성 및 보강

- 대화 요소 제거 ("당신", "네", 독자에게 던지는 질문 등)
- 한다 체로 통일, YAML 헤더 포함
- **단순 스타일 변환이 아니라** 배경 설명, 수식, 코드, 다이어그램 등을 보강
- 독자가 메모의 맥락을 모르더라도 포스트만으로 완결되도록 작성

### Step 6: index.qmd 업데이트

- 주 카테고리의 index.qmd에 새 포스트 링크를 추가한다
- Step 4에서 정한 **관련 카테고리의 index.qmd에도 크로스 링크를 추가**한다

### Step 7: TBD.qmd 초기화

전환 완료 후 빈 템플릿 상태로 되돌린다 (파일 삭제 금지).

```markdown
---
title: "TBD"
description: |
  임시 메모 보관. 주제별 정식 블로그 포스트로 전환 후 내용을 비운다.
draft: true
---
```

## 임시 파일 규칙

- 경로: `docs/blog/posts/TBD.qmd` (고정)
- 하나의 파일에 여러 주제를 누적하며, 주제 간 구분은 `---` 또는 `## 제목`
- YAML 헤더에 `draft: true`를 포함하여 빌드에서 제외
- 전환 후에도 파일은 삭제하지 않고 빈 상태로 유지
