# index.qmd 재구성 (Index Reorganization)

카테고리의 index.qmd가 포스트 증가로 인해 비체계적으로 된 경우,
포스트 메타데이터를 전수 분석하여 논리적 그룹핑과 순서를 재설계한다.

> **선행 조건**: info-search가 필요 없다. 대상 카테고리의 index.qmd와 GUIDE.md를 직접 읽는 것으로 충분하다.

---

## 작업 범위 결정

사용자 요청에 따라 작업 범위가 달라진다. 아래 표를 먼저 확인하고 적절한 모드를 선택한다.

| 사용자 입력 예시 | 모드 | 작업 범위 |
|----------------|------|----------|
| "Agent index.qmd 정리해줘" | **전체 모드** | 해당 카테고리 index.qmd 전체를 대상으로 한다 |
| "Agent/02-Prompt 정리해줘" | **섹션 모드** | 해당 서브폴더에 대응하는 index.qmd 내 섹션만 대상으로 한다 |
| "Agent Architecture 섹션 정리해줘" | **섹션 모드** | index.qmd에서 해당 섹션(## / ###) 헤더를 찾아 그 범위만 대상으로 한다 |

### 섹션 모드 동작 방식

사용자가 **서브폴더 경로** 또는 **섹션 이름**을 지정하면 섹션 모드로 진입한다.

1. **서브폴더 경로가 주어진 경우** (예: `Agent/02-Prompt`)
   - 상위 카테고리의 index.qmd를 전체 읽는다
   - 해당 서브폴더 경로(`./02-Prompt/`)를 포함하는 링크가 어느 ## / ### 섹션에 속하는지 특정한다
   - 해당 섹션(과 하위 섹션)만을 Step 2~7의 작업 범위로 한정한다
   - 서브폴더 내의 모든 .qmd 파일을 탐색하여 orphan을 탐지한다

2. **섹션 이름이 주어진 경우** (예: "Prompt Engineering 섹션")
   - index.qmd에서 해당 이름과 일치하는 ## / ### 헤더를 찾는다
   - 해당 헤더부터 다음 동급 헤더까지를 작업 범위로 한정한다

3. **범위 외 항목 보존**: 작업 범위 밖의 index.qmd 내용은 **한 글자도 변경하지 않는다**

::: {.callout-note}
## 섹션 모드에서도 전체 index.qmd를 읽는 이유

작업 범위가 특정 섹션이더라도, 진단 시 다른 섹션과의 관계(중복 배치, 크로스 참조)를 확인해야 한다. 따라서 Step 1에서 index.qmd 전체를 읽되, Step 2~7에서는 지정된 섹션만 수정한다.
:::

## Step 1: 패턴 식별 및 GUIDE.md 읽기

### 1-1. 카테고리 GUIDE.md 읽기

대상 카테고리의 `GUIDE.md`를 전체 읽는다. 특히 다음을 확인한다:

- 순차적 학습 경로가 정의되어 있는가
- 하위 주제 분류 체계가 있는가
- 교재 참조 구조가 있는가

### 1-2. index.qmd 전체 읽기 및 패턴 판별

index.qmd를 전체 읽고, AGENT_GUIDE.md에 정의된 4가지 패턴 중 어느 것인지 판별한다.

| 패턴 | 특징 | 판별 기준 |
|------|------|----------|
| A: 번호 목록 | `1.  [텍스트](경로)` | 번호로 시작, 날짜 없음, 섹션(###)으로 그룹핑 |
| B: 날짜 + bullet | `* YYYY-MM-DD, [텍스트](경로)` | bullet + 날짜가 선행 |
| C: 순수 목록 | 링크 있는/없는 항목 혼재 | 번호 없이 bullet, 일부 항목에 링크 없음 |
| D: 구조화된 학습 경로 | 설명 문단 + 중첩 bullet + placeholder | 섹션 간 설명 문단 존재, 1111-11-11 항목 다수 |

**패턴을 특정할 수 없는 경우**: 기존 항목 3개를 인용하며 사용자에게 확인한다 (Stop-and-Ask).

### Step 1 출력 형식 (반드시 출력)

```
## 패턴 판별 결과

- **카테고리**: (카테고리명)
- **index.qmd 총 줄 수**: (N줄)
- **판별된 패턴**: (A/B/C/D)
- **판별 근거**: (기존 항목 2-3개 인용)
- **기존 섹션 구조**: (## / ### 헤더 목록)
```

## Step 2: 포스트 메타데이터 전수 수집

### 2-1. 링크 추출

**전체 모드**: index.qmd에 등장하는 모든 .qmd/.ipynb 링크를 추출한다.
**섹션 모드**: 지정된 섹션 범위 내의 링크만 추출한다 (다른 섹션의 링크는 수집하되 "범위 외"로 표시).

각 링크에 대해:

- 상대경로를 절대경로로 변환
- 링크 텍스트를 기록
- placeholder 여부(1111-11-11 또는 0000-00-00)를 표시
- 크로스 카테고리 링크 여부(`../` 포함)를 표시

### 2-2. 각 .qmd 파일의 YAML 메타데이터 읽기

링크된 모든 .qmd 파일에 대해 **YAML 헤더만** 읽는다 (본문은 읽지 않는다).
수집 항목:

- `title`
- `subtitle` (있는 경우)
- `description`
- `categories` (배열)
- `date`
- 파일 존재 여부

### 2-3. 폴더 내 미등록 파일 탐지

**전체 모드**: 해당 카테고리 폴더와 하위 폴더를 탐색하여 .qmd 파일 중 index.qmd에 링크되지 않은 파일을 찾는다.
**섹션 모드**: 지정된 서브폴더 내의 .qmd 파일만 탐색하여 orphan을 탐지한다.

**제외 대상**: index.qmd, GUIDE.md, _metadata.yml

### 2-4. YAML 오류 탐지

수집한 YAML 메타데이터에서 다음 오류를 탐지한다:

| 오류 유형 | 탐지 기준 | 예시 |
|----------|----------|------|
| **title 불일치** | 파일명의 주제 키워드와 YAML title이 명백히 다름 | 파일명 `chain-of-thought.qmd`인데 title이 "Few-Shot Prompting" |
| **YAML 복붙 오류** | 연속된 2개 이상 파일의 title, subtitle, description이 동일 | 34, 35, 36번 파일이 모두 같은 title |
| **description 불일치** | description이 다른 파일의 내용을 설명 | 파일명 `directional-stimulus`인데 description이 "Active-Prompt의 정의..." |
| **필수 필드 누락** | title, description, categories, author, date 중 누락 | date가 없는 파일 |

오류가 발견되면 진단 보고서의 **D7 항목**으로 보고한다. 파일 본문의 첫 `## 헤더`와 첫 문단을 읽어 올바른 제목/설명을 추정하고, 사용자 승인 후 수정한다.

### 2-5. 카테고리 소속 적절성 판단

각 포스트의 YAML `categories` 배열과 내용이 **현재 폴더의 주제**와 일치하는지 판단한다.

| 판단 기준 | 설명 |
|----------|------|
| categories에 현재 카테고리 태그가 없음 | 예: `02-Prompt/`에 있는데 categories에 "Prompt Engineering"이 없고 "Agent, Architecture"만 있음 |
| 다른 폴더에 더 적합한 기존 그룹 존재 | 예: Agent Architecture 관련 포스트가 이미 `24-Agent-Architecture/`에 시리즈로 존재 |
| title/description이 현재 폴더의 주제보다 다른 주제를 주로 다룸 | 예: 프롬프트를 다루지만 본질이 "스킬 아키텍처 설계"인 경우 |

부적절 소속이 발견되면 진단 보고서의 **D8 항목**으로 보고하고, 이동 대상 폴더를 권장한다.

::: {.callout-important}
## 파일 이동은 사용자 승인 필수

카테고리 소속이 부적절하다고 판단되더라도, 파일 이동은 **AGENT_GUIDE.md의 Stop-and-Ask 규칙**(파일 이동 1개라도 승인 필요)을 따른다. 진단 보고서에서 이동 권장 목록을 제시하고, 사용자 승인 후에만 이동한다.
:::

### 2-6. 수집 결과 테이블 (반드시 출력)

```
| # | 파일명 | title | date | categories | 현재 섹션 | 상태 |
|---|--------|-------|------|------------|----------|------|
```

**상태 정의**:

| 상태 | 의미 |
|------|------|
| 정상 | index에 링크 있고 파일도 존재 |
| orphan | 파일은 존재하나 index에 링크 없음 |
| dead | index에 링크 있으나 파일 없음 (placeholder가 아닌 것) |
| placeholder | 1111-11-11 또는 0000-00-00 날짜, 아직 작성 안 된 계획 항목 |
| duplicate | 같은 파일이 index에 2회 이상 등장 |
| yaml-error | YAML 메타데이터에 오류(title 불일치, 복붙 등) |
| misplaced | 현재 폴더/카테고리에 소속이 부적절 |

## Step 3: 진단 보고서 출력

::: {.callout-important}
## 변경 전 필수 출력

아래 6가지 진단 항목을 **모두** 평가하고 결과를 출력한다.
진단 보고서를 출력하지 않고 Step 4로 넘어가면 안 된다.
:::

### 진단 항목

| # | 항목 | 평가 기준 | 심각도 |
|---|------|----------|--------|
| D1 | **Orphan 포스트** | 폴더에 존재하지만 index에 없는 .qmd | 높음 |
| D2 | **Dead 링크** | index에 링크 있으나 파일 없음 (placeholder 제외) | 높음 |
| D3 | **Duplicate 항목** | 동일 파일이 index에 2회 이상 등장 | 중간 |
| D4 | **주제 분산** | 같은 주제의 포스트가 서로 다른 섹션에 흩어져 있음 | 중간 |
| D5 | **순서 부적절** | 심화 포스트가 기초 포스트보다 앞에 위치 | 중간 |
| D6 | **섹션 불균형** | 특정 섹션에 포스트가 20개 이상 집중 | 낮음 |
| D7 | **YAML 오류** | title 불일치, 복붙 오류, 필수 필드 누락 | 높음 |
| D8 | **카테고리 부적절 소속** | 포스트가 현재 폴더/카테고리 주제와 맞지 않아 다른 폴더로 이동이 권장됨 | 중간 |

### 주제 유사도 판단 기준

포스트 간 주제 유사도는 다음 순서로 판단한다:

1. **YAML categories 겹침**: categories 배열에 공통 태그가 있으면 관련 주제
2. **title/subtitle 키워드 겹침**: 제목에 공통 키워드가 있으면 관련 주제
3. **description 의미 유사성**: description의 핵심 개념이 겹치면 관련 주제
4. **물리적 경로 공통성**: 같은 서브폴더에 있는 파일은 이미 그룹핑된 것

### 순서 적절성 판단 기준

동일 그룹 내 포스트의 순서는 다음 우선순위로 판단한다:

| 우선순위 | 기준 | 적용 조건 |
|---------|------|----------|
| 1 | **명시적 번호** | 파일명에 01-, 02- 등 번호가 있으면 그 순서를 존중 |
| 2 | **난이도 흐름** | title/description에서 기초->중급->고급 흐름을 판단 |
| 3 | **선후 의존** | 포스트 A가 포스트 B의 개념을 전제하면 A가 먼저 |
| 4 | **시간 순서** | 위 기준으로 판단 불가 시 date 오름차순 |

### 난이도 키워드 참고표

| 수준 | 자주 등장하는 키워드 |
|------|-------------------|
| 기초 | intro, overview, 개요, basics, 기초, 입문, 정의, what is, 개념 |
| 중급 | implementation, 구현, 활용, 패턴, design, 비교, 분석 |
| 고급 | advanced, optimization, production, 고급, 확장, scalability, 한계 |

### 진단 보고서 출력 형식

```
## 진단 보고서: [카테고리명] index.qmd

### 요약
- 작업 모드: 전체 / 섹션 (대상: [섹션명 또는 서브폴더])
- 총 포스트 수: N개 (정상 M, placeholder P, orphan O, dead D, duplicate U)
- 현재 섹션 수: S개
- 패턴: (A/B/C/D)

### D1. Orphan 포스트 (N개)
| 파일 | title | date | 권장 배치 섹션 |
|------|-------|------|--------------|

### D2. Dead 링크 (N개)
| index 내 위치(줄) | 링크 텍스트 | 경로 | 조치 권장 |
|------------------|-----------|------|----------|

### D3. Duplicate 항목 (N건)
| 파일 | 등장 섹션들 | 권장 조치 |
|------|-----------|----------|

### D4. 주제 분산 (N건)
| 주제 키워드 | 현재 위치 | 권장: 한 섹션으로 통합 |
|------------|----------|---------------------|

### D5. 순서 부적절 (N건)
| 포스트 | 현재 위치 | 문제 | 권장 위치 |
|--------|----------|------|----------|

### D6. 섹션 불균형
| 섹션 | 포스트 수 | 권장 조치 |
|------|----------|----------|

### D7. YAML 오류 (N건)
| 파일 | 오류 유형 | 현재 값 | 추정 올바른 값 |
|------|----------|--------|--------------|

### D8. 카테고리 부적절 소속 (N건)
| 파일 | 현재 폴더 | categories 태그 | 권장 이동 대상 | 이동 사유 |
|------|----------|---------------|-------------|----------|

### 종합 판단
- 재구성 필요도: 높음 / 중간 / 낮음
- 주요 문제: (1~2문장 요약)
```

## Step 4: 재구성안 설계 및 사용자 승인

### 4-1. 재구성 원칙 (절대 제약)

1. **index.qmd만 수정**: 이 스킬은 index.qmd의 목차 구조만 변경한다. **.qmd 파일의 이동, 폴더 생성/삭제, 파일 내용 수정은 일절 하지 않는다**
2. **패턴 보존**: Step 1에서 판별한 패턴(A/B/C/D)을 변경하지 않는다
3. **크로스 카테고리 링크 보존**: `../`로 시작하는 링크를 삭제하거나 이동하지 않는다
4. **Placeholder 보존**: 1111-11-11/0000-00-00 항목을 삭제하지 않는다. 위치 이동만 허용
5. **섹션 헤더 수동 번호 금지**: `## 1. 섹션명` 형태 금지
6. **References 섹션 보존**: 기존 References/참고 자료 섹션이 있으면 마지막에 유지
7. **링크 텍스트 보존**: `[...]` 안의 내용을 임의로 변경하지 않는다
8. **헤더 계층 존중**: 현재 ## -> ### -> #### 계층이 의미 있으면 유지. 불필요한 깊이 증가 금지

### 4-2. 세부 그룹핑 방법

하나의 섹션에 포스트가 많아(15개 이상) 탐색이 어려울 때, **하위 헤더(#### 또는 #####)를 추가**하여 논리 그룹을 만든다. 파일을 다른 폴더로 이동하거나 새 폴더를 생성하지 않는다.

**Before** (한 섹션에 43개 나열):

```
### Prompt Engineering
1.  [01-PromptTemplate](./02-Prompt/01-PromptTemplate.qmd)
2.  [02-FewShotPrompt](./02-Prompt/02-FewShotPrompt.qmd)
...
43. [43-system-prompt-evaluation](./02-Prompt/43-system-prompt-evaluation.qmd)
```

**After** (하위 헤더로 논리 그룹 분리):

```
### Prompt Engineering
```

## Step 5: 재구성 적용

사용자 승인을 받은 후 아래 순서로 적용한다.

### 5-0. 파일 이동 (D8 항목이 있고 사용자가 승인한 경우)

카테고리 부적절 소속으로 판단된 파일을 이동한다.

1. 대상 파일을 권장 폴더로 이동한다
2. **파일명 리넘버링**: 이동 대상 폴더의 기존 번호 체계에 맞춰 파일명의 번호를 변경한다 (예: `40-OldName.qmd` -> `19-new-name.qmd`)
3. **원본 index.qmd**에서 해당 파일의 링크를 제거한다
4. **이동 대상 카테고리의 index.qmd**에 새 파일명으로 링크를 추가한다 (대상 index의 패턴에 맞춰 작성)
5. **참조 링크 전수 업데이트**: 이동된 파일을 참조하는 모든 다른 포스트의 상대경로를 새 경로로 일괄 수정한다. Grep으로 구 파일명을 전체 검색하여 누락 없이 처리한다
6. 이동된 파일 내부의 상대경로 참조(이미지, 다른 포스트 링크 등)가 있으면 경로를 조정한다

### 5-0b. YAML 오류 수정 (D7 항목이 있고 사용자가 승인한 경우)

YAML 복붙 오류, title 불일치 등을 수정한다.

1. 파일 본문의 첫 `##` 헤더와 첫 문단을 읽어 올바른 title/description을 확인한다
2. YAML 헤더의 해당 필드를 수정한다
3. 수정한 파일 목록과 변경 내용을 기록한다

### 적용 규칙

1. **YAML 헤더 보존**: index.qmd의 YAML 헤더를 변경하지 않는다
2. **패턴별 포맷 엄수**:
   - 패턴 A: `N.  [텍스트](경로)` -- 각 섹션 내에서 1부터 재번호
   - 패턴 B: `* YYYY-MM-DD, [텍스트](경로)` -- 날짜 형식 유지
   - 패턴 C: `* [텍스트](경로)` 또는 `* 텍스트` -- 링크 유무 원본 그대로
   - 패턴 D: 설명 문단 + 중첩 bullet -- 기존 설명 문단 보존
3. **빈 줄 규칙**: 기존 index.qmd의 섹션 간 빈 줄 패턴을 유지
4. **Dead 링크**: 파일이 없고 placeholder도 아닌 경우 해당 줄 삭제 또는 사용자 지정 조치
5. **Orphan 추가**: YAML date를 읽어 적절한 형식으로 링크 작성, 주제 유사성에 따라 배치

### 대형 index 처리 (300줄 이상)

한 번에 전체를 재작성하면 누락 위험이 높다:

1. **섹션 단위 작업**: ## 헤더 기준으로 섹션을 분리하고 한 섹션씩 처리
2. **메타데이터 읽기 병렬화**: 서브폴더 단위로 YAML을 일괄 읽기
3. **변경 최소화**: 이미 잘 정리된 섹션은 건드리지 않는다. 진단에서 문제가 발견된 섹션만 재구성

## Step 6: 검증

재구성 완료 후 다음을 확인한다.

| 검증 항목 | 확인 내용 |
|----------|----------|
| 링크 전수 보존 | 재구성 전 정상 링크 수 = 재구성 후 링크 수 (orphan 추가분 제외) |
| Placeholder 보존 | 1111-11-11/0000-00-00 항목이 모두 존재하는가 |
| 크로스 링크 보존 | `../` 링크가 모두 존재하는가 |
| 패턴 일관성 | 모든 항목이 해당 패턴(A/B/C/D) 형식을 따르는가 |
| 헤더 수동 번호 | ## / ### 헤더에 수동 번호가 없는가 |
| YAML 헤더 무변경 | index.qmd의 YAML 헤더가 변경되지 않았는가 |
| 상대경로 정확성 | 링크의 상대경로가 올바른가 |

검증 실패 시 해당 항목을 수정한 후 다시 검증한다.
