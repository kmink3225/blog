# 시니어 레벨 해설 프레임워크 (면접관 방어용)

> `algo-tutor.md`와 `sql-tutor.md`에서 공유하는 하위 가이드이다.
> 모범 답안 제시 후 반드시 이 프레임워크에 따라 해설을 추가한다.

이 해설의 목적은 면접관 앞에서 **"왜 이 풀이/쿼리를 선택했는가"**를 논리적으로 방어할 수 있는 역량을 키우는 것이다.

## 알고리즘 (Python) 해설 구조

### 1. Big-O 복잡도 정밀 분석

단순히 "O(n)이다"로 끝내지 않는다. 아래를 포함한다:

- **Worst / Average / Best Case** 구분: 입력 분포에 따라 실제 성능이 달라지는 경우 명시한다
- **상수 계수(constant factor)와 실전 성능**: 점근적 복잡도가 같아도 캐시 지역성, 분기 예측 등으로 실제 속도가 다를 수 있다
- **Amortized 분석**: dict의 해시 충돌, 동적 배열 resize 등이 평균적으로 O(1)인 이유를 설명한다
- **공간-시간 트레이드오프**: O(n) 공간을 써서 O(n^2) → O(n)을 달성한 것의 정당성

```
예: "dict 조회는 해시 충돌이 없을 때 O(1)이다. Python dict는 open addressing으로
구현되어 있어 load factor 2/3 이하에서 평균 O(1)을 보장한다. 최악 O(n)이지만
랜덤 입력 가정 시 amortized O(1)이다. 따라서 전체 시간 복잡도는 average O(n),
worst O(n^2)이며, 실전에서 worst case가 발생할 확률은 극히 낮다."
```

### 2. 대안 풀이 비교 (Trade-off 분석) — 순위·Big O 필수

최소 2개 이상의 접근법을 비교하되, **반드시 아래 표 구조**를 따른다. `순위`, `시간 복잡도`, `공간 복잡도`, `순위 이유` 컬럼을 생략하지 않는다.

| 순위 | 접근법 | 시간 복잡도 | 공간 복잡도 | 장점 | 단점 | 순위 이유 |
|------|--------|-----------|-----------|------|------|----------|
| 1위 | Hash Map | 평균 O(n) / 최악 O(n²) | O(n) | 단일 패스, 조회 O(1) | 해시 충돌 최악 대비 | 입력 크기 제한 내 기대 성능 최적, 현실적으로 충돌 최악이 발생하지 않음 |
| 2위 | 정렬 + 투포인터 | O(n log n) | O(n) | 성능 안정적, 최악도 예측 가능 | 원본 인덱스 보존 필요 시 추가 구조 | 점근적으로 1위보다 느리지만 최악 보장 필요 시 선호 |
| 3위 | 브루트포스 | O(n²) | O(1) | 구현 간단, 메모리 ≈0 | n > 10^4에서 TLE | 점근적으로 가장 느리나 n ≤ 10^3에서는 상수 계수 덕에 1·2위보다 빠를 수 있음 |

**작성 규칙**:

- **순위**: 1위부터 N위까지 빠짐없이. 공동 순위는 `1위 (공동)`으로 표기하고 동점 사유를 순위 이유에 명시
- **시간·공간 복잡도**: 단일 표기 어려우면 `평균 / 최악`을 슬래시로 병기. 특수 가정(정렬된 입력, 균등 분포 등)은 괄호로 표시
- **순위 이유**: 측정 가능한 근거(점근적 상위, 상수 계수, 캐시 지역성, 메모리 한계 내 수용 여부)를 1~2문장으로. "직관적이다" 같은 주관만으로 순위 결정 금지

**실제 코드 블록 (각 순위마다 필수)**:

표 바로 아래에 **표에 등재한 모든 순위의 접근법마다 실행 가능한 코드를 ```` ```python ```` 블록으로 반드시 제시**한다. 표와 코드는 짝이며, 코드 없이 표만 제공하면 해설 미완료다.

- 각 코드 블록 위에 `### 1위 — {접근법 이름}` 같은 헤더로 순위·이름을 명시
- 코드 아래에 핵심 포인트·복잡도 근거를 2~4개 bullet로 짧게 덧붙인다 (무엇이 이 접근을 해당 순위로 만드는지)
- "코드가 너무 길어서 생략" 금지. 공통 입력 처리가 있으면 `# (입력 파싱 생략)` 주석으로 축약 가능하나 **순위 차이를 만드는 핵심 로직은 반드시 전체 표기**
- 반면교사(최하위) 코드도 생략 금지 — 왜 TLE/MLE인지 코드로 보여야 학습 효과가 있다

**해설 블록 (코드 묶음 아래 1~2문단 필수)**:

모든 코드 블록을 제시한 뒤 **"이 문제의 경우 X위가 최선인 이유"** 해설을 1~2문단으로 붙인다. 입력 크기, 메모리 제약, 분포 가정이 순위에 어떻게 영향을 주는지, 다른 조건이면 순위가 어떻게 뒤집히는지 적는다.

```
예: "본 문제의 제약은 n ≤ 10^5, 메모리 256MB다. 이 범위에서는 1위(Hash Map)가
점근적으로 가장 빠르고 메모리도 충분히 수용된다. n이 10^9으로 커져 메모리가 부족해지면
2위(정렬 + 투포인터)로 외부 정렬(External Sort)과 결합한 변형이 현실적 선택이 된다.
3위(브루트포스)는 n ≤ 10^3일 때만 상수 계수 덕에 1·2위보다 빠를 수 있으나, 본 문제에선
TLE가 확정되므로 면접 답안에선 언급만 하고 배제하는 방어 논리가 적절하다."
```

**완성된 대안 비교 블록의 구조 (순서 고정)**:

```
1. 비교 표 (순위·시간/공간 복잡도·장단점·순위 이유 컬럼)
2. 각 순위별 실제 Python 코드 (```python``` 블록 + 헤더 + 핵심 bullet)
3. "X위가 최선인 이유" 해설 1~2문단
```

세 구성요소 중 하나라도 빠지면 해설 미완료로 간주한다.

"면접관이 '왜 정렬 대신 Hash를 택했는가?' 물으면" 형식으로 방어 논리를 제시한다.

### 3. 통계적/확률론적 관점 (해당 시)

DS 트랙 또는 확률/통계가 관련된 문제에서는 아래를 포함한다:

- 입력 분포에 따른 기대 복잡도 (expected complexity under uniform distribution 등)
- 확률적 자료구조(Bloom Filter, Reservoir Sampling 등)의 오탐률/정확도 분석
- 몬테카를로 시뮬레이션 문제의 수렴 속도와 분산 분석

### 4. 공학적 실전 고려사항

면접에서 "실제 프로덕션이라면?"이라는 후속 질문에 대비한다:

- **메모리 제약**: n이 10^8 이상이면 O(n) 공간도 부담 — 스트리밍/외부 정렬 필요
- **병렬화 가능성**: 이 알고리즘을 MapReduce/멀티스레드로 확장할 수 있는가
- **캐시 효율**: 배열 순차 접근 vs 해시 랜덤 접근의 실제 성능 차이
- **엣지 케이스 방어**: 입력 검증, 오버플로우, 부동소수점 비교 등

```
예: "이 Hash 풀이는 메모리가 충분한 단일 머신 환경에서 최적이다.
만약 n이 메모리를 초과하면 외부 해시(External Hashing)나 정렬 기반
접근으로 전환해야 한다. 면접관에게 '제약 조건을 확인한 후 최적 전략을
선택했다'는 점을 보여주는 것이 중요하다."
```

### 5. 면접 방어 요약 (한 줄)

```
"n <= 10^5이고 메모리 제한이 충분하므로 O(n) 시간 + O(n) 공간의 Hash Map이
최적이다. 정렬 기반(O(n log n))도 가능하지만 상수 계수와 인덱스 보존 측면에서
Hash가 유리하다."
```

## SQL 해설 구조

### 1. 실행 계획(Execution Plan) 관점 분석

단순히 "정답이다"로 끝내지 않는다. 아래를 포함한다:

- **Full Table Scan vs Index Scan**: WHERE/JOIN 조건이 인덱스를 탈 수 있는 구조인가
- **JOIN 순서와 드라이빙 테이블**: 옵티마이저가 어떤 테이블을 먼저 읽을지, 왜 그것이 효율적인지
- **서브쿼리 vs JOIN vs CTE**: 같은 결과를 내는 여러 방식의 실행 비용 차이
- **정렬 비용**: ORDER BY, GROUP BY, DISTINCT가 유발하는 임시 테이블/파일 정렬

```
예: "이 쿼리는 employees 테이블을 dept_id로 GROUP BY한다. dept_id에
인덱스가 있으면 Index Scan + Streaming Aggregation으로 O(n)에 처리된다.
인덱스가 없으면 Full Scan 후 Hash Aggregation이 발생하여 임시 메모리를
O(그룹 수)만큼 사용한다."
```

### 2. 대안 쿼리 비교 (Trade-off 분석) — 순위·Big O 필수

최소 2개 이상의 접근법을 비교하되, **반드시 아래 표 구조**를 따른다. `순위`, `Big O 복잡도`, `순위 이유` 컬럼을 생략하지 않는다.

| 순위 | 접근법 | Big O 복잡도 | 장점 | 단점 | 적합한 상황 | 순위 이유 |
|------|--------|-------------|------|------|------------|----------|
| 1위 | JOIN + GROUP BY | O(n log n) (정렬 기반) / O(n) (Hash Aggregation) | 옵티마이저 최적화 유리, 단일 패스 집계 | 결과 중복 주의 | 대규모 테이블 | JOIN 키 인덱스 활용 시 단일 스캔으로 완료, 상관 서브쿼리보다 실행 횟수 훨씬 적음 |
| 2위 | Window Function | O(n log n) (파티션별 정렬) | 행 유지, 순위·누적 계산 자연스러움 | MySQL 8.0+ 필요, 메모리 사용량↑ | 순위·이동집계 | JOIN 집계와 점근적 동일하지만 정렬 강제로 상수 계수 큼 |
| 3위 | 상관 서브쿼리 | O(n × m) (외부 행마다 내부 반복) | 직관적, 표현력 높음 | 외부 행 수만큼 반복 실행 → 대용량 치명적 | 소규모 또는 상관 조건이 본질적인 경우 | 옵티마이저가 Semi/Anti-Join으로 변환 못 하면 원본 복잡도 그대로 — 최악 |

**작성 규칙**:

- **순위**: 1위부터 N위까지 빠짐없이. 공동 순위는 허용(예: `1위 (공동)`) — 이 경우 "동점 사유" 를 순위 이유에 명시
- **Big O 복잡도**: 단일 표기가 어려우면 `평균 O(n) / 최악 O(n²)` 처럼 상황별로 병기. 인덱스 가정이 필요하면 `(PK 인덱스 가정)` 같은 괄호 주석
- **순위 이유**: "왜 이 순위인가"를 **실행 계획 관점**에서 1~2문장으로. "가독성 좋음" 같은 주관만으로 순위 결정 금지 — 반드시 비용·스캔 횟수·메모리 등 측정 가능한 근거 포함
- **적합한 상황**: 해당 접근이 순위와 무관하게 유리해지는 전환점 명시 (예: "소규모 n < 1만", "인덱스 부재 시", "NULL 다수 포함")

**실제 쿼리 코드 블록 (각 순위마다 필수)**:

표 바로 아래에 **표에 등재한 모든 순위의 접근법마다 실행 가능한 쿼리 코드를 ```` ```sql ```` 블록으로 반드시 제시**한다. 표와 코드는 짝이며, 코드 없이 표만 제공하면 해설 미완료다.

- 각 쿼리 블록 위에 `### 1위 — {접근법 이름}` 같은 헤더로 순위·이름을 명시
- 쿼리 아래에 핵심 포인트·주의사항을 2~4개 bullet로 짧게 덧붙인다 (무엇이 이 접근을 1위/N위로 만드는지)
- "코드가 너무 길어서 생략" 금지. 중복되는 보일러플레이트가 있으면 `-- (공통 JOIN 생략)` 주석으로 축약 가능하나 **순위 차이를 만드는 핵심 구문은 반드시 전체 표기**
- 반면교사(최하위) 쿼리도 코드 생략 금지 — 왜 최악인지 코드로 보여야 학습 효과가 있다

**해설 블록 (코드 묶음 아래 1~2문단 필수)**:

모든 쿼리 코드 블록을 제시한 뒤 **"이 문제의 경우 X위가 최선인 이유"** 해설을 1~2문단으로 붙인다. 데이터 규모·인덱스 가정·방언 제약이 어떻게 순위에 영향을 주는지, 다른 조건이면 순위가 어떻게 뒤집히는지 적는다.

```
예: "본 문제의 PRODUCT 테이블은 수십만 행 수준이고 category_id에 인덱스가 있다는 전제다.
이 조건에서는 1위(JOIN + GROUP BY)가 압도적으로 유리하다 — Hash Aggregation이 O(n)에
수렴한다. 만약 PRODUCT가 수천만 행이고 인덱스가 없다면 Window Function(2위)의
정렬 비용이 문제 되므로, ROLAP 집계 테이블을 미리 만들어 두는 방식이 현업 대안이 된다.
상관 서브쿼리(3위)는 입력 크기에 무관하게 n이 조금만 커져도 치명적이므로 면접 답안으로
언급만 하고 실제 선택은 피한다는 서술이 방어 논리로 적절하다."
```

**완성된 대안 비교 블록의 구조 (순서 고정)**:

```
1. 비교 표 (순위·Big O·장단점·순위 이유 컬럼)
2. 각 순위별 실제 쿼리 코드 (```sql``` 블록 + 헤더 + 핵심 bullet)
3. "X위가 최선인 이유" 해설 1~2문단
```

세 구성요소 중 하나라도 빠지면 해설 미완료로 간주한다.

"면접관이 '서브쿼리 대신 JOIN으로 바꾸면 어떻게 되는가?' 물으면" 형식으로 방어 논리를 제시한다.

### 3. 통계적/데이터 공학적 관점 (해당 시)

데이터 특성이 쿼리 성능에 미치는 영향을 분석한다:

- **카디널리티(Cardinality)**: GROUP BY 컬럼의 고유값 수가 성능에 미치는 영향
- **데이터 분포와 skew**: 특정 값에 데이터가 편중되면 Hash JOIN의 메모리 사용량이 급증한다
- **NULL 비율**: LEFT JOIN + IS NULL 패턴에서 NULL 비율이 높으면 결과 집합이 커진다
- **인덱스 선택도(Selectivity)**: WHERE 조건의 선택도가 낮으면(대부분 행 통과) 인덱스보다 Full Scan이 유리하다

### 4. 공학적 실전 고려사항

면접에서 "프로덕션 환경이라면?"이라는 후속 질문에 대비한다:

- **대용량 테이블**: 수억 행 테이블에서 이 쿼리가 동작하는가 — 파티셔닝, 인덱스 전략
- **동시성(Concurrency)**: 이 쿼리가 락을 유발하는가, 읽기/쓰기 충돌 가능성
- **방언 호환**: MySQL vs PostgreSQL vs Oracle에서 동작 차이 (LIMIT vs ROWNUM, IFNULL vs COALESCE)
- **쿼리 캐시와 실행 빈도**: 자주 실행되는 쿼리라면 뷰(View) 또는 Materialized View 고려

```
예: "이 윈도우 함수 쿼리는 MySQL 8.0+에서만 동작한다.
MySQL 5.7 환경이라면 상관 서브쿼리로 대체해야 한다.
면접에서 DB 버전을 먼저 확인하는 습관을 보여주면 좋다."
```

### 5. 면접 방어 요약 (한 줄)

LEFT JOIN + IS NULL 대신 NOT EXISTS를 택한 이유는 옵티마이저가 Anti-Join으로
변환할 수 있어 대용량에서 유리하기 때문이다."

## 포스트 저장 시 면접 방어 포인트 기록

모범 답안의 면접 방어 요약은 포스트 저장 시 각 문제의 **배운 점** 섹션 아래에 **면접 방어 포인트**로 기록한다.

### 알고리즘 포스트 템플릿

```markdown
### 면접 방어 포인트
- **복잡도 정밀 분석**: worst/average/best case 구분, 상수 계수 논의
- **대안 비교**: 왜 이 접근법을 택했는가 (trade-off 근거)
- **공학적 확장**: 프로덕션 제약 시 어떻게 변형하는가
```

### SQL 포스트 템플릿

```markdown
### 면접 방어 포인트
- **실행 계획 분석**: 이 쿼리의 실행 비용과 인덱스 활용 여부
- **대안 비교**: 서브쿼리/JOIN/CTE 중 왜 이 접근을 택했는가 (trade-off 근거)
- **공학적 확장**: 대용량 테이블, 방언 호환, 동시성 고려사항
```
